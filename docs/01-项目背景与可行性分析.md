# 心屿 AI 情绪记录与匿名互助社区系统

# 01-项目背景与可行性分析

## 1. 项目基本信息

### 1.1 项目名称

心屿 AI 情绪记录与匿名互助社区系统

### 1.2 项目类型

本项目属于基于 Java Web 的前后端分离应用，融合了个人情绪记录、AI 情绪陪伴、匿名互助社区、后台内容管理和数据统计等能力。

从软件工程角度看，项目可以归类为：

1. 前后端分离 Web 应用
2. 用户内容管理系统
3. AI 应用系统
4. 轻量级社区平台
5. 心理健康辅助类工具系统

### 1.3 项目定位

心屿面向有情绪记录、压力表达和自我观察需求的用户，提供一个安全、私密、温和的线上情绪空间。用户可以记录每日心情、查看历史情绪变化、获得 AI 温暖回复，也可以选择以匿名方式发布情绪帖子，在社区中获得评论、点赞和收藏等互动支持。

项目最终目标不是替代专业心理咨询，而是作为一个日常情绪记录与陪伴工具，帮助用户更好地表达情绪、整理想法和观察自身状态。

## 2. 项目背景

### 2.1 现实背景

在学习、工作、社交和生活压力不断增加的环境下，越来越多年轻用户存在情绪表达和情绪记录的需求。很多用户并不一定需要正式心理咨询，但需要一个低门槛、低压力、可持续使用的空间来记录情绪、表达烦恼和获得陪伴。

传统的备忘录或日记应用主要解决“记录”问题，但缺少及时反馈；普通社区平台可以提供互动，但用户往往担心身份暴露、评价压力和内容不够私密。AI 大模型的发展让系统可以根据用户输入生成更自然、更温和的回复，为情绪记录类应用提供了新的交互方式。

因此，设计一个集“私人心情记录 + AI 回复 + 匿名互助社区 + 情绪统计 + 后台管理”于一体的系统，具有较好的实践价值和应用扩展价值。

### 2.2 项目建设背景

从项目完整性角度看，仅实现简单的登录注册和单表 CRUD 功能，系统业务复杂度和应用价值相对有限。心屿项目可以在基础 CRUD 之上引入登录认证、权限控制、社区互动、AI 调用、数据统计、后台审核等业务场景，使系统更接近真实中小型 Web 应用。

该项目通过模块化设计覆盖以下常见 Web 应用开发能力：

1. Spring Boot 接口开发
2. MyBatis-Plus 数据访问
3. MySQL 表设计与关联查询
4. JWT 登录认证
5. BCrypt 密码加密
6. 统一返回结果与全局异常处理
7. 前后端分离与跨域配置
8. Vue3 页面开发与 axios 请求封装
9. 社区类业务设计
10. 管理端审核与数据统计
11. AI API 接入与结果持久化
12. Redis、分页、条件查询、部署等扩展能力

这些内容能够支撑较完整的系统建设，也能为后续功能扩展提供清晰的技术基础。

## 3. 项目建设目标

### 3.1 总体目标

建设一个基于 Spring Boot、Vue3、MySQL、Redis 和大模型 API 的 AI 情绪记录与匿名互助社区系统，实现用户登录、心情记录、AI 情绪回复、匿名发帖、评论点赞收藏、后台内容审核、情绪统计和社区数据统计等功能。

系统既要满足用户端的情绪记录与社区互动需求，也要提供管理员端的内容治理能力，使项目具备完整平台型应用的结构。

### 3.2 用户端目标

用户端主要服务普通用户，目标包括：

1. 用户可以注册、登录和退出登录。
2. 用户可以新增、查看、修改和删除自己的心情记录。
3. 用户新增心情记录后，可以获得 AI 温暖回复和简单情绪建议。
4. 用户可以查看自己的历史心情记录和情绪变化趋势。
5. 用户可以匿名发布帖子，表达自己的情绪和想法。
6. 用户可以浏览社区帖子，进行评论、点赞和收藏。
7. 用户可以根据话题分类查看内容，如学习压力、实习焦虑、人际关系、睡眠问题等。

### 3.3 管理端目标

管理端主要服务平台管理员，目标包括：

1. 管理用户账号状态，如启用、禁用和搜索用户。
2. 管理帖子内容，如审核、拒绝、删除和按状态筛选。
3. 管理评论内容，如查看评论列表和删除违规评论。
4. 管理社区话题，如新增、修改、禁用话题。
5. 查看 AI 回复记录，便于检查 AI 输出内容和调用情况。
6. 查看系统统计数据，如用户数、帖子数、评论数、AI 调用次数、热门话题等。

### 3.4 AI 能力目标

AI 助手不是系统中的真实用户，而是系统提供的一种智能服务能力。AI 能力目标包括：

1. 根据用户心情记录生成温暖回复。
2. 根据用户帖子内容生成简单陪伴式评论。
3. 根据用户一段时间内的记录生成情绪总结。
4. 对明显负面或高风险内容进行简单风险提示。
5. 生成每日鼓励语，提高系统陪伴感。

## 4. 系统角色分析

### 4.1 普通用户

普通用户是系统的核心使用者，主要需求是记录情绪、获得反馈、匿名表达和查看自我变化。

普通用户可以使用的功能包括：

1. 注册和登录。
2. 修改个人资料，如昵称、头像和密码。
3. 新增心情记录。
4. 查看心情记录列表和详情。
5. 修改或删除自己的心情记录。
6. 获取 AI 情绪回复。
7. 发布匿名帖子。
8. 查看帖子列表和帖子详情。
9. 评论帖子。
10. 点赞帖子。
11. 收藏帖子。
12. 查看情绪统计图表。

### 4.2 管理员

管理员负责系统内容治理和基础运营管理。管理员不参与用户私人情绪记录的日常使用，而是管理公开社区内容和平台基础数据。

管理员可以使用的功能包括：

1. 管理用户。
2. 审核帖子。
3. 审核或删除评论。
4. 管理话题分类。
5. 查看 AI 回复记录。
6. 查看社区数据统计。
7. 查看情绪数据统计。
8. 处理违规内容。

### 4.3 AI 助手

AI 助手不是一个真实登录角色，而是系统中的智能服务模块。它通过大模型 API 接收用户输入内容，并生成回复结果。

AI 助手可以完成：

1. 心情记录回复生成。
2. 社区帖子温暖评论生成。
3. 情绪总结生成。
4. 简单风险提示。
5. 每日鼓励语生成。

## 5. 核心业务模块规划

系统规划为六个核心模块：

1. 用户与权限模块
2. 情绪记录模块
3. AI 回复模块
4. 匿名社区模块
5. 后台管理模块
6. 数据统计模块

### 5.1 用户与权限模块

该模块是系统基础模块，负责用户身份、登录认证和权限控制。

主要功能：

1. 用户注册。
2. 用户登录。
3. JWT 登录认证。
4. 退出登录。
5. 查询当前用户信息。
6. 修改昵称。
7. 修改头像。
8. 修改密码。
9. 管理员登录。
10. 基于角色的权限控制。

核心技术点：

1. Spring Boot
2. MyBatis-Plus
3. MySQL
4. JWT
5. 拦截器
6. BCrypt 密码加密
7. 统一返回 Result
8. 全局异常处理

### 5.2 情绪记录模块

该模块是心屿项目的核心私人记录功能，用户可以长期记录和回顾自己的情绪变化。

主要功能：

1. 新增心情记录。
2. 查看心情记录列表。
3. 查看心情记录详情。
4. 修改心情记录。
5. 删除心情记录。
6. 按日期查询。
7. 按心情类型查询。
8. 分页查询。
9. 设置记录是否公开。

设计说明：

心情记录可以包含 mood_type、mood_score、content、ai_reply、is_public 等字段。其中 is_public 用于区分记录是私人日记，还是可以同步发布到匿名社区，从而让情绪记录模块和社区模块产生自然关联。

### 5.3 AI 回复模块

AI 回复模块是项目区别于普通 CRUD 系统的重要亮点。该模块负责调用大模型 API，并保存 AI 回复记录。

主要功能：

1. 用户新增心情后生成 AI 回复。
2. 用户发布帖子后生成 AI 温暖评论。
3. 保存 AI 回复记录。
4. 查询 AI 回复历史。
5. 管理员查看 AI 回复记录。
6. 记录模型名称、输入内容和输出结果。

设计说明：

AI 回复模块建议独立设计，通过 target_type 和 target_id 关联不同业务场景。例如 target_type 可以是 MOOD_RECORD、POST 或 CHAT，target_id 对应具体心情记录、帖子或聊天消息。这样 AI 回复能力不会写死在某一个业务模块中，后续扩展性更好。

设计说明：

AI 回复模块通过 target_type 和 target_id 关联不同业务场景，比如心情记录、社区帖子和聊天消息，从而提高模块复用性和扩展性。

### 5.4 匿名社区模块

匿名社区模块用于提升系统业务复杂度和互动性，是项目从“个人记录工具”升级为“平台型应用”的关键。

主要功能：

1. 发布匿名帖子。
2. 查看帖子列表。
3. 查看帖子详情。
4. 删除自己的帖子。
5. 评论帖子。
6. 删除自己的评论。
7. 点赞帖子。
8. 收藏帖子。
9. 按话题分类查看。
10. 按热度排序。
11. 按发布时间排序。

社区话题示例：

1. 学习压力
2. 实习焦虑
3. 人际关系
4. 情绪低落
5. 开心分享
6. 睡眠问题

### 5.5 后台管理模块

后台管理模块用于体现系统的完整性和内容治理能力。该模块不需要过度复杂，但应具备基本管理能力。

主要功能：

1. 用户管理：查看用户列表、禁用用户、启用用户、按用户名搜索。
2. 帖子管理：查看帖子列表、审核帖子、删除帖子、按状态筛选。
3. 评论管理：查看评论列表、删除评论、按帖子查询评论。
4. 话题管理：新增话题、修改话题、禁用话题。
5. AI 回复记录：查看 AI 调用记录、查看失败记录、查看用户输入和 AI 回复。
6. 数据统计：查看用户、帖子、评论、AI 调用等统计数据。

### 5.6 数据统计模块

数据统计模块用于增强用户对情绪变化和社区运行情况的理解，并体现 SQL 聚合查询、数据可视化和缓存等能力。

用户端统计：

1. 最近 7 天心情趋势。
2. 最近 30 天心情趋势。
3. 不同心情类型占比。
4. 心情平均分。
5. 记录次数统计。

管理端统计：

1. 用户总数。
2. 今日新增用户。
3. 帖子总数。
4. 今日新增帖子。
5. 评论总数。
6. AI 调用次数。
7. 热门话题排行。

可体现技术点：

1. SQL 聚合查询。
2. 分组统计。
3. ECharts 图表。
4. Redis 缓存统计结果。

## 6. 核心业务流程

### 6.1 用户记录心情并获得 AI 回复

流程说明：

1. 用户登录系统。
2. 用户填写心情类型、心情分数和心情内容。
3. 前端提交心情记录。
4. 后端校验 token，识别当前用户。
5. 后端保存心情记录。
6. 后端调用 AI 接口。
7. AI 根据心情内容生成温暖回复。
8. 后端保存 AI 回复记录。
9. 前端展示 AI 回复。

该流程是心屿项目最核心的业务闭环。

### 6.2 用户发布匿名社区帖子

流程说明：

1. 用户登录系统。
2. 用户选择话题。
3. 用户编写帖子标题和内容。
4. 用户选择是否匿名。
5. 前端提交帖子。
6. 后端保存帖子。
7. 系统根据审核策略将帖子设置为待审核或正常展示。
8. 帖子出现在社区列表。

### 6.3 用户评论、点赞和收藏

流程说明：

1. 用户进入帖子详情页。
2. 用户查看帖子内容。
3. 用户发表评论。
4. 用户点赞帖子。
5. 用户收藏帖子。
6. 系统更新评论数、点赞数和收藏状态。

该流程体现一对多、多对多关系和用户行为数据记录。

### 6.4 管理员审核内容

流程说明：

1. 管理员登录后台。
2. 管理员查看待审核帖子。
3. 管理员进入帖子详情。
4. 管理员选择审核通过、拒绝或删除。
5. 后端更新帖子状态。
6. 前端刷新审核列表。

该流程体现后台管理、角色权限和内容状态流转。

核心实体关系如下：

1. user 1 对 N mood_record
2. user 1 对 N post
3. user 1 对 N comment
4. user 1 对 N post_like
5. user 1 对 N favorite

9. post 1 对 N favorite

该关系设计相比单纯 CRUD 项目更丰富，适合体现数据库建模和业务分析能力。

## 8. 可行性分析

### 8.1 技术可行性

项目采用的技术栈成熟稳定，适合中小型前后端分离 Web 应用开发。

后端方面，Spring Boot 能够快速搭建 REST API，MyBatis-Plus 能够简化数据库 CRUD 和分页查询，JWT 可以实现无状态登录认证，BCrypt 可以保证密码加密存储。前端方面，Vue3、Vue Router 和 axios 能够完成页面开发、路由跳转和接口联调。MySQL 能够满足用户、帖子、评论、心情记录等结构化数据存储需求。

AI 回复功能可以通过调用大模型 API 实现，前期可以先完成同步调用和保存结果，后续再考虑异步任务、失败重试和调用日志。

因此，从技术角度看，本项目具备可实现性。

### 8.2 经济可行性

本项目开发阶段可以使用本地开发环境完成，无需高成本服务器。数据库可以使用本地 MySQL，前端和后端均可本地运行。

AI 接口调用可能产生少量成本，但可以通过限制调用次数、使用测试额度或先使用模拟回复降低成本。项目后续部署时，可以选择低成本云服务器或免费托管平台进行演示。

因此，从经济角度看，本项目成本可控。

### 8.3 时间可行性

项目可以分阶段实现，避免一次性开发过多功能导致失控。

建议开发阶段如下：

1. 第一阶段：最小闭环。完成注册、登录、JWT、心情记录新增、心情记录列表、AI 回复。
2. 第二阶段：社区闭环。完成发帖、帖子列表、帖子详情、评论、点赞、收藏。
3. 第三阶段：后台管理。完成管理员登录、用户管理、帖子管理、评论管理、帖子审核。
4. 第四阶段：统计与扩展能力。完成情绪统计、社区统计、Redis 缓存、文件上传头像、部署上线和接口文档。

通过分阶段开发，可以保证每个阶段都有可演示成果。

### 8.4 操作可行性

系统面向普通用户，核心操作包括记录心情、查看记录、浏览帖子、发表评论等，交互逻辑清晰，学习成本较低。

管理员端主要是列表管理、状态修改和内容审核，操作模式接近常见后台管理系统，易于理解和实现。

因此，从用户操作角度看，本项目具有较好的可用性。

### 8.5 维护可行性

系统采用模块化设计，将用户、心情记录、AI 回复、社区、后台管理和统计模块分开。后端采用 Controller、Service、Mapper 分层结构，前端按页面和请求工具进行组织。

这种结构便于后续维护和扩展。例如，AI 回复模块可以在不影响心情记录基础 CRUD 的情况下扩展；社区模块也可以独立增加点赞、收藏和话题功能。

因此，从维护角度看，本项目具备较好的扩展性。

## 9. 项目特色与应用价值

### 9.1 项目创新点

与普通后台管理系统或简单 CRUD 项目相比，心屿项目具有以下特点：

1. 主题明确：围绕情绪记录和情绪陪伴展开，而不是泛泛的管理系统。
2. AI 结合自然：AI 回复与心情记录场景结合紧密，有真实业务意义。
3. 匿名社区扩展：在私人记录基础上增加社区互动，提高业务复杂度。
4. 管理端治理：加入帖子审核、评论管理和用户管理，使项目更完整。
5. 数据统计展示：通过情绪趋势和社区统计增强系统的数据表达能力。
6. 安全设计较完整：包含 JWT、密码加密、统一异常和敏感字段保护。

### 9.2 工程实践价值

相比普通“登录注册 + 单表 CRUD”系统，本项目能够体现更完整的 Web 应用开发能力：

1. 业务建模能力：能设计用户、心情记录、帖子、评论、点赞、收藏、AI 回复等实体关系。
2. 接口设计能力：能设计 RESTful 风格接口和统一响应结构。
3. 权限认证能力：能使用 JWT 实现登录认证和接口权限控制。
4. 数据库设计能力：能处理一对多、多对多、状态字段、统计查询等场景。
5. 前后端联调能力：能使用 Vue3、axios 和 request 拦截器完成 token 携带。
6. 安全意识：能说明为什么不返回 password，为什么不信任前端 userId。
7. AI 应用能力：能接入大模型 API 并保存 AI 回复记录。
8. 平台治理能力：能设计后台审核和内容状态管理。

### 9.3 项目说明示例

项目名称：

心屿 AI 情绪记录与匿名互助社区系统

项目描述：

心屿是一个基于 Spring Boot、Vue3、MySQL、Redis 和大模型 API 的 AI 情绪记录与匿名互助社区系统，支持用户登录、心情记录、AI 情绪回复、匿名发帖、评论点赞收藏、内容审核、情绪统计和后台管理等功能。

技术栈：

Spring Boot、MyBatis-Plus、MySQL、Redis、Vue3、axios、JWT、ECharts、大模型 API、Nginx

项目亮点：

1. 使用 JWT + 拦截器实现登录认证和接口权限控制。
2. 使用 MyBatis-Plus 实现心情记录、帖子、评论等模块的分页与条件查询。
3. 接入大模型 API，根据用户心情内容生成 AI 情绪陪伴回复，并保存回复记录。
4. 设计匿名社区模块，支持发帖、评论、点赞、收藏和话题分类。
5. 设计后台审核模块，对帖子和评论进行状态管理，提高平台内容安全性。
6. 使用 ECharts 展示用户情绪趋势和社区数据统计。
7. 使用 Redis 缓存热门帖子、首页统计或 AI 回复结果，提高接口响应效率。

## 10. 风险分析与应对方案

### 10.1 功能范围过大风险

风险说明：

项目规划包含用户、心情记录、AI、社区、后台和统计等多个模块，如果一次性全部开发，容易导致周期过长、功能混乱。

应对方案：

采用分阶段开发方式，先完成最小闭环，再逐步扩展社区、后台和统计功能。每个阶段都保证可运行、可演示。

### 10.2 AI 接口不稳定风险

风险说明：

AI 接口可能出现调用失败、响应慢、额度不足或结果不稳定等问题。

应对方案：

1. 保存 AI 调用记录。
2. 对失败情况返回友好提示。
3. 前期可以使用模拟回复保证主流程可用。
4. 后续增加失败重试和异步处理。

### 10.3 内容安全风险

风险说明：

匿名社区可能出现违规内容、恶意评论或不适当表达。

应对方案：

1. 帖子和评论增加 status 状态。
2. 管理员可以审核、拒绝和删除内容。
3. 后续可接入敏感词过滤。
4. AI 对高风险内容进行简单提示。

### 10.4 权限控制风险

风险说明：

如果后端直接使用前端传入的 userId，用户可能篡改参数访问其他用户数据。

应对方案：

1. 后端从 token 中解析 userId。
2. 用户只能查询和操作自己的数据。
3. 删除、修改接口增加所属用户校验。
4. 管理员接口增加角色校验。
