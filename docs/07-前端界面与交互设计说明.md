# 心屿 AI 情绪记录与匿名互助社区系统

# 07-前端界面与交互设计说明

## 1. 文档目的

本文档用于说明心屿 AI 情绪记录与匿名互助社区系统的前端界面设计和交互设计方案，包括整体视觉风格、页面结构、用户操作流程、组件规划、状态反馈、异常提示、响应式设计和后续扩展方向。

前端设计以产品体验为出发点，同时结合软件工程中的组件化、可维护性和前后端联调规范，保证页面既能清晰表达业务，也便于后续功能扩展。

## 2. 前端设计目标

前端设计需要满足以下目标：

1. 提供清晰、低压力的情绪记录体验。
2. 让用户能够快速完成注册、登录、记录心情、查看记录和社区互动。
3. 通过温和的视觉风格强化系统的陪伴感。
4. 通过统一组件和请求封装降低维护成本。
5. 支持用户端和管理端两类页面。
6. 支持后续 AI 回复、情绪统计、匿名社区和后台审核扩展。

## 3. 整体视觉风格

### 3.1 产品气质

心屿的产品气质应以“温和、安静、可信、低压力”为主。界面不应给用户强烈压迫感，也不应使用过于商业化或刺激性的视觉表达。

建议关键词：

1. 温和。
2. 安静。
3. 陪伴感。
4. 私密感。
5. 轻量。
6. 清晰。

### 3.2 色彩风格

用户端建议使用浅色背景和柔和辅助色。

推荐方向：

1. 主色：柔和蓝、浅紫或蓝紫色。
2. 背景色：接近白色或极浅灰蓝。
3. 成功色：柔和绿色。
4. 警告色：柔和橙色。
5. 危险色：低饱和红色。
6. 文本色：深灰为主，减少纯黑带来的压迫感。

### 3.3 页面氛围

用户端页面应避免过于复杂的布局，优先使用卡片、留白、圆角和柔和阴影，让用户能够专注于表达情绪。

管理端页面应更偏工具化，使用清晰的表格、筛选栏、状态标签和操作按钮，重点保证管理效率。

## 5. 路由设计

### 5.1 用户端路由

| 路由 | 页面 | 是否需要登录 | 说明 |
| --- | --- | --- | --- |
| / | Login.vue | 否 | 登录页 |
| /register | Register.vue | 否 | 注册页 |
| /home | Home.vue | 是 | 首页，记录与查看心情 |
| /mood/{id} | MoodDetail.vue | 是 | 心情详情页 |
| /posts | PostList.vue | 否 | 社区列表页 |
| /posts/{id} | PostDetail.vue | 否 | 帖子详情页 |
| /post/publish | PostPublish.vue | 是 | 发布帖子页 |
| /profile | Profile.vue | 是 | 个人中心 |
| /mood/stat | MoodStat.vue | 是 | 情绪统计页 |

### 5.2 管理端路由

| 路由 | 页面 | 是否需要管理员 |
| --- | --- | --- |
| /admin/login | AdminLogin.vue | 否 |
| /admin/home | AdminHome.vue | 是 |
| /admin/users | UserManage.vue | 是 |
| /admin/posts | PostManage.vue | 是 |
| /admin/comments | CommentManage.vue | 是 |
| /admin/topics | TopicManage.vue | 是 |
| /admin/ai-replies | AiReplyManage.vue | 是 |
| /admin/stat | AdminStat.vue | 是 |

### 5.3 路由守卫设计

后续可使用 router.beforeEach 统一处理登录状态：

```text
进入需要登录的页面
-> 检查 localStorage 中是否存在 token
-> 不存在则跳转登录页
-> 存在则允许进入
```

管理员页面还需要校验 role：

```text
进入管理员页面
-> 检查 token
-> 检查 loginUser.role 是否为 ADMIN
-> 否则跳转无权限页或登录页
```

## 6. 登录页设计

### 6.1 页面目标

登录页用于用户输入用户名和密码，完成登录并进入系统首页。

### 6.2 页面元素

1. 项目名称。
2. 简短说明文案。
3. 用户名输入框。
4. 密码输入框。
5. 登录按钮。
6. 注册入口。
7. 错误提示。

### 6.3 交互流程

```text
用户输入用户名和密码
-> 点击登录
-> 前端校验是否为空
-> 调用 /user/login
-> 登录成功，保存 loginUser 和 token
-> 跳转 /home
-> 登录失败，显示错误提示
```

### 6.4 状态设计

| 状态 | 表现 |
| --- | --- |
| 初始状态 | 输入框为空，按钮可点击 |
| 输入中 | 用户输入账号密码 |
| 提交中 | 登录按钮显示“登录中...”并禁用 |
| 成功 | 跳转首页 |
| 失败 | 展示错误提示 |

## 7. 注册页设计

### 7.1 页面目标

注册页用于新用户创建账号。

### 7.2 页面元素

1. 用户名输入框。
2. 密码输入框。
3. 昵称输入框。
4. 注册按钮。
5. 返回登录入口。

### 7.3 交互流程

```text
用户填写注册信息
-> 点击注册
-> 前端校验字段是否为空
-> 调用 /user/register
-> 注册成功，提示成功并跳转登录页
-> 注册失败，展示失败原因
```

### 7.4 校验规则

1. 用户名不能为空。
2. 密码不能为空。
3. 昵称不能为空。
4. 密码长度可限制为 6 到 20 位。

## 8. 首页设计

### 8.1 页面目标

首页用于承载当前阶段的核心闭环：欢迎用户、新增心情记录、查看心情记录列表、删除记录和退出登录。

### 8.2 页面结构

首页建议分为三块：

1. 顶部用户区域。
2. 新增心情记录区域。
3. 我的心情记录列表区域。

### 8.3 顶部用户区域

展示内容：

1. 项目名称。
2. 欢迎语。
3. 当前用户昵称。
4. 退出登录按钮。

交互：

1. 点击退出登录。
2. 清除 localStorage 中的 loginUser 和 token。
3. 跳转登录页。

### 8.4 新增心情记录区域

表单字段：

1. 心情类型。
2. 心情内容。
3. 后续可增加心情分数。
4. 后续可增加是否公开。

交互流程：

```text
用户输入心情类型和内容
-> 点击新增
-> 校验是否为空
-> 调用新增接口
-> 新增成功后清空表单
-> 自动刷新记录列表
```

### 8.5 心情记录列表区域

展示内容：

1. 心情类型标签。
2. 心情内容。
3. 记录时间。
4. 删除按钮。
5. 后续可增加 AI 回复摘要。

交互：

1. 点击刷新，重新请求列表。
2. 点击删除，弹出确认提示。
3. 删除成功后刷新列表。
4. 点击记录卡片，后续可进入详情页。

### 8.6 空状态设计

当用户暂无心情记录时，页面不应只显示空白，而应展示友好提示。

示例：

```text
暂无心情记录
写下第一条记录，给今天的情绪一个出口。
```

### 8.7 心情标签设计

不同心情类型可使用不同标签样式：

| 心情类型 | 样式方向 |
| --- | --- |
| 开心 | 柔和蓝紫 |
| 平静 | 柔和绿色 |
| 焦虑 | 柔和橙色 |
| 难过 | 灰蓝色 |
| 其他 | 默认浅紫色 |

## 17. 交互状态设计

### 17.1 加载状态

接口请求过程中应展示加载状态，避免用户重复点击。

示例：

1. 登录按钮显示“登录中...”。
2. 新增按钮显示“提交中...”。
3. 列表区域显示 loading。

### 17.2 成功状态

操作成功后应给出明确反馈。

示例：

1. 注册成功。
2. 登录成功。
3. 新增成功。
4. 删除成功。
5. 发布成功。

### 17.3 失败状态

操作失败时应展示后端返回的 message。

示例：

1. 用户名已存在。
2. 账号或密码错误。
3. 请先登录。
4. 权限不足。
5. 服务异常，请稍后再试。

### 17.4 确认状态

危险操作前应二次确认。

适用场景：

1. 删除心情记录。
2. 删除帖子。
3. 删除评论。
4. 禁用用户。

## 18. 响应式设计

### 18.1 桌面端

桌面端可以使用双栏布局：

1. 左侧为新增心情区域。
2. 右侧为心情记录列表。

社区页面可以使用列表 + 侧边话题栏。

管理端使用表格布局。

### 18.2 移动端

移动端应使用单栏布局：

1. 表单区域和列表区域上下排列。
2. 按钮宽度适当增大。
3. 卡片间距适当缩小。
4. 表格类管理页面可横向滚动或简化展示字段。

## 20. 前端设计结论

心屿系统前端设计以“温和、安静、低压力、陪伴感”为用户端核心风格，以“清晰、效率、工具化”为管理端核心风格。用户端重点保证记录心情、查看历史、获得 AI 回复和社区互动的体验流畅；管理端重点保证内容审核、用户管理和数据统计的操作效率。

通过 Vue3 页面组件化、Vue Router 路由管理、request.js 请求封装、localStorage 登录状态保存、空状态和加载状态设计，前端能够支撑当前最小闭环，也能为后续匿名社区、AI 回复、情绪统计和后台管理扩展提供基础。
