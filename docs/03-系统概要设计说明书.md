# 心屿 AI 情绪记录与匿名互助社区系统

# 03-系统概要设计说明书

## 1. 文档目的

本文档用于说明心屿 AI 情绪记录与匿名互助社区系统的总体架构、技术选型、模块划分、系统角色、数据流转方式和安全设计思路。

概要设计位于需求分析之后、详细设计之前，主要回答以下问题：

1. 系统整体采用什么架构。
2. 前端、后端、数据库和 AI 服务如何协作。
3. 系统由哪些核心模块组成。
4. 用户端和管理端分别包含哪些功能。
5. 登录认证、权限控制、异常处理和统一返回如何设计。

## 2. 系统总体架构

### 2.1 架构模式

系统采用前后端分离架构。

前端使用 Vue3 构建用户界面，负责页面展示、表单输入、路由跳转和接口调用。后端使用 Spring Boot 提供 REST API，负责业务处理、身份校验、数据库读写、AI 调用和统一响应。数据库使用 MySQL 保存用户、心情记录、帖子、评论、点赞、收藏、AI 回复等业务数据。

系统后续可接入 Redis，用于缓存热门帖子、统计数据、验证码或临时 token 信息；可接入大模型 API，用于生成 AI 情绪回复和情绪总结。

### 2.2 架构组成

系统主要由以下部分组成：

| 组成部分 | 技术选型 | 主要职责 |
| --- | --- | --- |
| 用户端前端 | Vue3、Vue Router、axios | 登录注册、心情记录、社区互动、情绪统计展示 |
| 管理端前端 | Vue3、Element Plus、axios | 用户管理、帖子审核、评论管理、话题管理、数据统计 |
| 后端服务 | Spring Boot、MyBatis-Plus | 提供 REST API、处理业务逻辑、权限校验 |
| 数据库 | MySQL | 保存系统核心业务数据 |
| 缓存服务 | Redis | 缓存热点数据、统计结果和临时数据 |
| AI 服务 | 大模型 API | 生成 AI 回复、情绪总结和风险提示 |

### 2.3 系统访问流程

普通用户访问流程：

1. 用户打开前端页面。
2. 用户注册或登录。
3. 后端校验账号密码。
4. 登录成功后后端生成 JWT token。
5. 前端保存 loginUser 和 token。
6. 前端后续请求通过 request.js 自动携带 token。
7. 后端解析 token，识别当前用户。
8. 后端根据用户身份处理心情记录、帖子、评论等业务。

管理员访问流程：

1. 管理员进入后台登录页。
2. 管理员登录成功后获取 token。
3. 前端访问管理接口时携带 token。
4. 后端解析 token 并校验角色是否为 ADMIN。
5. 校验通过后允许访问用户管理、帖子审核、评论管理和统计接口。

AI 回复流程：

1. 用户新增心情记录或发布帖子。
2. 后端保存用户提交内容。
3. 后端组装 prompt，调用大模型 API。
4. AI 返回回复结果。
5. 后端保存 AI 回复记录。
6. 前端展示 AI 回复内容。

## 3. 技术选型设计

### 3.1 后端技术选型

| 技术 | 说明 |
| --- | --- |
| Java 17 | 项目后端开发语言 |
| Spring Boot | 用于快速构建 Web 服务和 REST API |
| MyBatis-Plus | 简化数据库 CRUD、分页和条件查询 |
| MySQL | 保存结构化业务数据 |
| JWT | 实现无状态登录认证 |
| BCrypt | 用户密码加密存储 |
| Knife4j | 基于 springdoc-openapi 的接口文档与调试界面 |
| Redis | 后续用于缓存统计数据、热门内容和临时数据 |
| Maven | 后端项目依赖管理和构建工具 |

### 3.2 前端技术选型

| 技术 | 说明 |
| --- | --- |
| Vue3 | 构建前端页面和组件，使用 `<script setup>` 组合式 API |
| Vue Router | 实现页面路由跳转和登录守卫 |
| Pinia | 全局状态管理，维护用户、心情、社区等跨页面状态 |
| axios | 发送 HTTP 请求 |
| request.js | 对 axios 进行二次封装，统一携带 token、统一处理响应 |
| Vite | 前端开发和构建工具 |
| ECharts | 后续用于展示情绪趋势和统计图表 |

说明：前端未使用第三方 UI 组件库，自建一套设计 token 与基础组件（按钮、卡片、输入、弹窗、标签、分页、空状态、骨架屏等），保证视觉风格统一可控。

### 3.3 技术选型理由

Spring Boot、MyBatis-Plus、MySQL 和 Vue3 是 Java Web 项目中常见且成熟的组合，适合中小型前后端分离系统。JWT 和 BCrypt 用于支撑基础安全设计；Knife4j 提供可视化接口文档便于联调；Pinia 在前端统一管理跨页面状态；Redis、ECharts 和 AI API 则用于后续增强系统的数据处理、可视化和智能回复能力。

## 4. 系统模块划分

系统按业务职责划分为六个核心模块：

1. 用户与权限模块
2. 情绪记录模块
3. AI 回复模块
4. 匿名社区模块
5. 后台管理模块
6. 数据统计模块

## 5. 用户与权限模块概要设计

### 5.1 模块职责

用户与权限模块负责用户注册、登录、退出登录、个人信息维护、密码加密、JWT token 生成与校验、用户角色判断等功能。

### 5.2 核心功能

1. 用户注册。
2. 用户登录。
3. 管理员登录。
4. 退出登录。
5. 查询当前用户信息。
6. 修改用户资料。
7. 修改密码。
8. JWT 登录认证。
9. 管理员权限校验。

### 5.3 设计要点

1. 用户注册时密码必须使用 BCrypt 加密后再保存。
2. 用户登录成功后生成 JWT token。
3. token 中保存 userId、username 和过期时间。
4. 前端保存 token，后续请求自动携带。
5. 后端从 token 中解析 userId，不信任前端传入的 userId。
6. 返回给前端的用户信息使用 UserVO，不包含 password。
7. 管理员接口需要校验 role 是否为 ADMIN。

## 6. 情绪记录模块概要设计

### 6.1 模块职责

情绪记录模块负责用户个人心情记录的新增、查询、详情、修改、删除和筛选统计，是系统用户端的核心功能模块。

### 6.2 核心功能

1. 新增心情记录。
2. 查询心情记录列表。
3. 查看心情记录详情。
4. 修改心情记录。
5. 删除心情记录。
6. 按日期查询。
7. 按心情类型查询。
8. 分页查询。
9. 设置记录是否公开。

### 6.3 设计要点

1. 所有心情记录接口都需要登录。
2. 后端通过 token 解析当前用户 id。
3. 用户只能查询、修改和删除自己的记录。
4. 新增记录时由后端设置 userId，而不是前端传入 userId。
5. 心情记录可以与 AI 回复关联。
6. 心情记录可以通过 is_public 字段决定是否公开到社区。

## 7. AI 回复模块概要设计

### 7.1 模块职责

AI 回复模块负责调用大模型 API，根据用户输入生成情绪陪伴回复、帖子温暖评论、情绪总结和风险提示，并保存 AI 调用结果。

### 7.2 核心功能

1. 根据心情记录生成 AI 回复。
2. 根据帖子内容生成 AI 评论。
3. 保存 AI 回复记录。
4. 查询 AI 回复历史。
5. 管理员查看 AI 调用记录。
6. 生成阶段性情绪总结。

### 7.3 设计要点

1. AI 回复模块应独立设计，不直接绑定单一业务表。
2. 使用 target_type 和 target_id 关联不同业务场景。
3. target_type 可以包括 MOOD_RECORD、POST、CHAT 等。
4. AI 调用失败不能影响基础业务保存。
5. AI 回复内容需要保存，便于用户查看和管理员审查。

## 8. 匿名社区模块概要设计

### 8.1 模块职责

匿名社区模块负责用户公开表达、帖子浏览、评论互动、点赞收藏和话题分类，是系统从个人工具扩展为互动平台的重要模块。

### 8.2 核心功能

1. 发布匿名帖子。
2. 查看帖子列表。
3. 查看帖子详情。
4. 删除自己的帖子。
5. 评论帖子。
6. 删除自己的评论。
7. 点赞帖子。
8. 取消点赞。
9. 收藏帖子。
10. 取消收藏。
11. 按话题查看帖子。
12. 按热度和时间排序。

### 8.3 设计要点

1. 用户必须登录后才能发帖、评论、点赞和收藏。
2. 帖子支持 anonymous 字段，决定前端是否展示真实用户信息。
3. 帖子使用 status 字段支持审核和删除状态。
4. 点赞和收藏需要防止同一用户重复操作。
5. 评论和帖子都需要支持管理员审核或删除。

## 9. 后台管理模块概要设计

### 9.1 模块职责

后台管理模块负责平台内容治理和基础数据管理，主要服务管理员。

### 9.2 核心功能

1. 管理员登录。
2. 用户管理。
3. 帖子管理。
4. 评论管理。
5. 话题管理。
6. AI 回复记录查看。
7. 平台数据统计。

### 9.3 设计要点

1. 管理接口必须校验登录状态。
2. 管理接口必须校验用户角色为 ADMIN。
3. 帖子、评论通过 status 字段完成状态流转。
4. 管理端页面以表格、筛选、分页和操作按钮为主。
5. 后台不追求复杂视觉效果，优先保证功能清晰。

## 10. 数据统计模块概要设计

### 10.1 模块职责

数据统计模块负责用户情绪数据和平台运营数据的统计分析，为用户端和管理端提供可视化展示。

### 10.2 用户端统计

1. 最近 7 天心情趋势。
2. 最近 30 天心情趋势。
3. 不同心情类型占比。
4. 平均心情分数。
5. 记录次数统计。

### 10.3 管理端统计

1. 用户总数。
2. 今日新增用户。
3. 帖子总数。
4. 今日新增帖子。
5. 评论总数。
6. AI 调用次数。
7. 热门话题排行。
8. 高互动帖子排行。

### 10.4 设计要点

1. 统计数据通过 SQL 聚合查询实现。
2. 前端使用 ECharts 展示图表。
3. 热门数据和统计结果可使用 Redis 缓存。
4. 统计接口应支持时间范围参数。

## 11. 前端概要设计

### 11.1 用户端页面

当前已实现的用户端页面：

1. 登录页。
2. 注册页。
3. 首页（心情记录与最近 7 天趋势，新增心情和列表在同一页）。
4. 帖子详情页（含回复列表与发表回复）。
5. 社区列表页（发帖表单内嵌于本页）。
6. 个人收藏页。
7. 个人资料页（修改昵称、修改密码）。

说明：心情详情、心情编辑通过首页的弹窗组件实现，未单独成页；情绪统计页（最近 30 天趋势、占比图表）属于后续数据统计阶段。

### 11.2 管理端页面

管理端页面包括：

1. 管理员登录页。
2. 后台首页。
3. 用户管理页。
4. 帖子管理页。
5. 评论管理页。
6. 话题管理页。
7. AI 回复记录页。
8. 数据统计页。

### 11.3 前端设计原则

用户端风格应温和、轻量、低压力，突出陪伴感和记录感。管理端风格应简洁、清晰、偏工具化，重点保证列表筛选、状态管理和数据查看效率。

### 11.4 前端请求设计

前端使用 request.js 封装 axios：

1. 统一设置 baseURL。
2. 统一设置请求超时时间。
3. 请求拦截器自动读取 localStorage 中的 token 并放入请求头。
4. 响应拦截器统一处理登录失效，并为网络错误附加可读提示。

在 request.js 之上，前端再分三层组织：

1. api 层：按业务域（user、mood、community、topic）封装具体接口，统一拆包后端响应。
2. stores 层：使用 Pinia 管理用户、心情、社区等跨页面状态。
3. composables 层：封装通用逻辑，如全局提示、确认弹窗、异步动作与分页。

页面只负责组装组件和触发 store 动作，不直接调用 axios。

### 11.5 前端组件与样式设计

前端不引入第三方 UI 组件库，自建设计系统：

1. styles：全局设计 token（颜色、字号、间距、圆角、阴影、动效）与基础样式。
2. 基础组件：按钮、卡片、输入框、文本域、下拉、标签、弹窗、分页、空状态、骨架屏等。
3. 布局组件：页面外壳、顶部导航、页脚。
4. 业务组件：心情卡片、心情表单、趋势面板、帖子卡片、回复列表等。

## 12. 后端概要设计

### 12.1 后端分层结构

后端采用常见分层结构：

| 层级 | 职责 |
| --- | --- |
| Controller | 接收请求、参数校验、返回 Result |
| Service | 处理业务逻辑 |
| Mapper | 操作数据库 |
| Entity | 对应数据库表 |
| DTO | 接收前端请求参数 |
| VO | 返回前端展示数据 |
| Utils | 通用工具，如 JWT 工具类 |
| Config | 系统配置，如 CORS、密码加密 |
| Exception | 全局异常处理 |

### 12.2 统一返回设计

后端接口统一返回 Result：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

统一返回可以降低前端处理复杂度，使所有接口都可以通过 code、message、data 进行统一判断。

### 12.3 统一异常设计

系统使用 GlobalExceptionHandler 统一处理异常。业务中未捕获的异常会被统一转换为 Result.error，避免后端异常堆栈暴露给前端。

### 12.4 跨域设计

由于前端和后端运行在不同端口，系统通过 CORS 配置允许前端地址访问后端接口，并允许常用请求方法和请求头。

## 13. 安全概要设计

### 13.1 密码安全

用户密码不允许明文存储。注册时使用 BCrypt 加密密码，登录时使用 matches 方法校验密码。

### 13.2 登录认证

系统使用 JWT 实现登录认证。登录成功后，后端生成 token 并返回给前端。前端保存 token，后续请求自动携带 token。后端解析 token 后识别当前用户。

### 13.3 权限控制

普通用户只能访问用户端接口，管理员可以访问后台管理接口。管理员接口需要校验用户角色。

### 13.4 数据隔离

用户只能访问自己的心情记录、收藏记录和个人信息。后端从 token 中解析 userId，不依赖前端传入 userId，防止用户篡改参数访问他人数据。

### 13.5 敏感字段保护

后端返回用户信息时使用 VO 对象，不返回 password 等敏感字段。

## 14. 数据存储概要设计

系统核心数据表规划如下：

| 表名 | 说明 |
| --- | --- |
| user | 用户表 |
| mood_record | 心情记录表 |
| ai_reply | AI 回复记录表 |
| post | 社区帖子表 |
| comment | 评论表 |
| post_like | 点赞表 |
| favorite | 收藏表 |
| topic | 话题表 |

核心关系：

1. 一个用户可以有多条心情记录。
2. 一个用户可以发布多篇帖子。
3. 一个帖子可以有多条评论。
4. 一个帖子可以被多个用户点赞。
5. 一个帖子可以被多个用户收藏。
6. 一个话题下可以有多篇帖子。
7. AI 回复可以关联心情记录或帖子。

## 15. 部署概要设计

### 15.1 本地开发环境

本地开发阶段：

1. 前端运行在 localhost:5173。
2. 后端运行在 localhost:8080。
3. MySQL 运行在 localhost:3306。
4. 前端通过 axios 请求后端接口。
5. 后端通过 JDBC 连接 MySQL。

### 15.2 后续部署方案

后续可采用以下部署方式：

1. 前端通过 Nginx 部署静态资源。
2. 后端通过 jar 包运行在服务器。
3. MySQL 部署在服务器或云数据库。
4. Redis 部署在服务器或云缓存。
5. 使用 Nginx 反向代理后端接口。

## 16. 系统概要设计结论

心屿 AI 情绪记录与匿名互助社区系统采用前后端分离架构，后端基于 Spring Boot 和 MyBatis-Plus，前端基于 Vue3，数据库使用 MySQL，并规划接入 Redis 和大模型 API。

系统从功能上划分为用户与权限、情绪记录、AI 回复、匿名社区、后台管理和数据统计六个模块。该设计既能满足当前最小闭环开发，也能支撑后续扩展为完整的平台型 Web 应用。

通过 JWT 登录认证、BCrypt 密码加密、统一 Result、全局异常处理、CORS 配置、VO 返回对象和后端解析 userId 等设计，系统具备较好的规范性、安全性和可扩展性。
