# 心屿 AI 情绪记录与匿名互助社区系统

# 05-接口设计说明书

## 1. 文档目的

本文档用于说明心屿 AI 情绪记录与匿名互助社区系统的后端接口设计方案，包括接口设计原则、统一响应格式、认证方式、请求规范、错误处理规则，以及用户、心情记录、AI 回复、匿名社区、后台管理和数据统计等模块的接口规划。

接口设计以产品业务流程为出发点，同时结合 REST API 规范和软件工程可维护性要求，保证前后端联调清晰、模块边界明确、权限控制合理。

## 2. 接口设计目标

接口设计需要满足以下目标：

1. 支撑前端用户端和管理端页面调用。
2. 保持接口路径、请求方式和返回结构统一。
3. 使用 JWT token 进行登录认证。
4. 后端从 token 解析 userId，不依赖前端传入 userId 进行身份判断。
5. 返回数据不暴露 password 等敏感字段。
6. 支持分页、条件查询和状态筛选。
7. 支持后续 AI 回复、社区互动、后台审核和数据统计扩展。
8. 便于使用 Apifox、Postman 或 Swagger 进行接口测试。

## 3. 基础约定

### 3.1 服务地址

开发环境：

```text
后端服务地址：http://localhost:8080
前端服务地址：http://localhost:5173
```

### 3.2 请求数据格式

POST、PUT 请求默认使用 JSON：

```http
Content-Type: application/json
```

### 3.3 响应数据格式

所有接口统一返回 JSON。

### 3.4 时间格式

后端时间字段建议使用 ISO 日期时间格式：

```text
2026-05-14T20:30:12
```

前端展示时可转换为：

```text
2026-05-14 20:30
```

### 3.5 命名规范

1. 路径使用小写英文。
2. 多单词路径使用中划线或业务名词组合，避免过长。
3. 请求参数使用小驼峰格式，例如 moodType、moodScore。
4. 返回字段使用小驼峰格式。

## 4. 统一响应格式

### 4.1 Result 结构

系统所有业务接口统一返回 Result：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

### 4.2 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| code | number | 业务状态码，200 表示成功，500 表示业务失败 |
| message | string | 提示信息 |
| data | any | 响应数据 |

### 4.3 成功示例

```json
{
  "code": 200,
  "message": "查询成功",
  "data": []
}
```

### 4.4 失败示例

```json
{
  "code": 500,
  "message": "请先登录",
  "data": null
}
```

### 4.5 设计说明

统一响应格式可以降低前端处理成本。前端只需要根据 code 判断业务是否成功，再读取 message 和 data 进行提示或渲染。

## 5. 认证与权限设计

### 5.1 登录认证方式

系统使用 JWT token 维护登录状态。用户登录成功后，后端返回 token，前端保存 token。后续请求通过请求头携带 token。

当前请求头约定：

```http
token: xxxxx.yyyyy.zzzzz
```

后续也可扩展为更标准的 Authorization 方式：

```http
Authorization: Bearer xxxxx.yyyyy.zzzzz
```

### 5.2 token 使用流程

1. 用户调用登录接口。
2. 后端校验账号密码。
3. 后端生成 JWT token。
4. 前端保存 token 到 localStorage。
5. request.js 请求拦截器自动携带 token。
6. 后端从请求头读取 token。
7. 后端解析 token 得到 userId 和 username。
8. 后端根据 userId 处理当前用户的数据。

### 5.3 需要登录的接口

以下接口需要携带 token：

1. 心情记录相关接口。
2. AI 回复相关接口。
3. 社区发帖、评论、点赞、收藏接口。
4. 当前用户信息接口。
5. 后台管理接口。
6. 数据统计中与用户相关的接口。

### 5.4 管理员权限

管理员接口除校验 token 外，还需要校验用户角色为 ADMIN。

如果未登录：

```json
{
  "code": 500,
  "message": "请先登录",
  "data": null
}
```

如果无管理员权限：

```json
{
  "code": 500,
  "message": "权限不足",
  "data": null
}
```

## 7.2 用户登录

### 7.2.1 接口说明

用户输入用户名和密码登录系统，登录成功后返回用户信息和 token。

### 7.2.2 请求信息

| 项目 | 内容 |
| --- | --- |
| 请求方式 | POST |
| 请求路径 | /user/login |
| 是否需要 token | 否 |

### 7.2.3 请求参数

```json
{
  "username": "test",
  "password": "123456"
}
```

### 7.2.4 响应示例

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "user": {
      "id": 1,
      "username": "test",
      "nickname": "小屿",
      "avatar": null
    },
    "token": "xxxxx.yyyyy.zzzzz"
  }
}
```

### 7.2.5 设计说明

1. 返回 user 对象不包含 password。
2. token 由后端生成，用于后续接口认证。
3. 前端需要保存 loginUser 和 token。

## 17. 接口设计结论

本系统接口围绕用户与权限、心情记录、AI 回复、匿名社区、后台管理和数据统计六个业务方向进行设计。接口整体采用 REST 风格，使用统一 Result 返回结构，使用 JWT token 进行登录认证，并通过用户角色区分普通用户和管理员权限。

接口设计强调后端身份可信原则：用户相关数据操作应以后端解析 token 得到的 userId 为准，而不是直接信任前端传入的 userId。该设计能够提升系统安全性，并为后续社区互动、后台审核和数据统计功能扩展提供稳定基础。
