# 03-系统概要设计说明书

## 1. 文档目的

本文档用于说明“心屿 AI 情绪记录与匿名互助社区系统”的整体架构、技术选型、模块划分、数据流、安全设计和部署概要。

当前项目已经完成用户端、社区端、后台管理端和 AI 回复记录管理的基础闭环，因此本文档按当前实现状态进行描述，而不是停留在早期规划阶段。

## 2. 系统总体架构

系统采用前后端分离架构：

```text
浏览器
  -> Vue3 前端应用
  -> axios / request.js
  -> Spring Boot REST API
  -> MyBatis-Plus
  -> MySQL
  -> DeepSeek API
```

前端负责页面展示、表单输入、路由跳转、状态管理和接口调用。后端负责登录认证、权限判断、业务处理、数据库读写、AI 调用和统一响应。MySQL 保存用户、心情记录、社区内容、互动关系和 AI 调用日志。

## 3. 技术选型

| 层级 | 技术 | 说明 |
| --- | --- | --- |
| 前端 | Vue3、Vite、Pinia、Vue Router、axios | 构建用户端和后台管理端 |
| 后端 | Spring Boot、MyBatis-Plus、JWT、BCrypt | 提供 REST API、认证和业务逻辑 |
| 数据库 | MySQL | 保存核心业务数据 |
| AI | DeepSeek API | 生成心情和帖子 AI 回复 |
| 文档 | Knife4j、Markdown | 接口调试和项目说明 |
| 后续增强 | Redis、ECharts、Nginx | 缓存、图表和部署增强 |

前端没有依赖大型 UI 框架，主要使用自建基础组件和业务组件。后台管理页复用 `AdminTabs`、`AdminPageHeader`、`AdminFilterActions` 等组件保持风格统一。

## 4. 角色设计

### 4.1 普通用户

普通用户可以：

- 注册和登录
- 记录、查看、修改、删除自己的心情记录
- 查看 AI 回复和最近 7 天心情趋势
- 发布、修改、删除自己的社区帖子
- 回复帖子
- 点赞、收藏帖子
- 查看收藏列表

### 4.2 管理员

管理员可以：

- 进入后台概览
- 管理用户状态
- 管理帖子状态
- 管理回复状态
- 管理话题
- 查看 AI 回复记录
- 查看平台统计数据

管理员与普通用户共用登录体系，通过 `role=ADMIN` 区分权限，不单独设计管理员登录页。

## 5. 模块划分

系统按业务职责划分为六个模块：

| 模块 | 主要职责 |
| --- | --- |
| 用户与权限模块 | 注册、登录、JWT、角色权限、密码加密 |
| 心情记录模块 | 心情 CRUD、筛选分页、详情、最近 7 天趋势 |
| AI 回复模块 | 调用 DeepSeek、生成回复、失败兜底、日志保存 |
| 匿名社区模块 | 话题、帖子、回复、点赞、收藏 |
| 后台管理模块 | 用户、帖子、回复、话题、AI 记录管理 |
| 数据统计模块 | 用户端趋势、后台概览、AI 调用统计 |

## 6. 核心业务流程

### 6.1 登录认证流程

```text
用户提交用户名和密码
-> 后端校验账号和 BCrypt 密码
-> 登录成功后生成 JWT token
-> 前端保存 token 和用户信息
-> 后续请求由 request.js 自动携带 token
-> 后端解析 token 获取当前用户
```

### 6.2 心情记录 AI 回复流程

```text
用户新增或修改心情记录
-> 后端保存心情数据
-> 后端根据心情类型和内容组装 prompt
-> 调用 DeepSeek API
-> 将最新回复写入 mood_record.ai_reply
-> 将完整调用记录写入 ai_reply 表
-> 前端展示 AI 回复
```

如果 AI 调用失败，系统仍保存心情记录，并写入兜底回复和失败日志。

### 6.3 社区帖子 AI 回复流程

```text
用户发布或修改帖子
-> 后端保存帖子内容
-> 后端根据话题、情绪和正文生成 prompt
-> 调用 DeepSeek API
-> 将最新回复写入 community_post.ai_reply
-> 将调用记录写入 ai_reply 表，targetType=POST
-> 帖子详情页展示 AI 心屿回复
```

### 6.4 后台管理流程

```text
管理员登录
-> 前端进入 /admin 相关页面
-> 请求后台接口时携带 token
-> 后端校验用户角色是否为 ADMIN
-> 校验通过后执行用户、帖子、回复、话题或 AI 记录管理操作
```

普通用户访问后台接口会被拒绝。

### 7.3 前端状态管理

Pinia store 按业务拆分：

- `user`：登录态和用户信息
- `mood`：心情记录、筛选、分页、趋势
- `community`：帖子、详情、回复、点赞、收藏、话题
- `admin`：后台用户、帖子、回复、话题、AI 记录和概览统计

### 7.4 请求设计

前端使用 `request.js` 封装 axios：

- 统一配置 `baseURL`
- 请求自动携带 token
- 统一处理响应和错误
- api 层按业务域拆分
- 页面优先调用 store，不直接散落请求逻辑

## 8. 后端概要设计

后端采用常见分层结构：

| 层级 | 职责 |
| --- | --- |
| Controller | 接收请求、校验参数、返回 Result |
| Service | 处理业务逻辑 |
| Mapper | 操作数据库 |
| Entity | 对应数据库表 |
| DTO | 接收前端请求参数 |
| VO | 返回前端展示数据 |
| Config | CORS、密码加密等配置 |
| Exception | 全局异常处理 |
| Utils | JWT 等工具类 |

后端接口统一返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

统一返回可以降低前端处理复杂度，让页面只关心业务结果。

## 9. 数据存储概要

核心数据表：

| 表 | 说明 |
| --- | --- |
| `user` | 用户表 |
| `mood_record` | 心情记录表 |
| `ai_reply` | AI 回复调用记录表 |
| `community_topic` | 社区话题表 |
| `community_post` | 社区帖子表 |
| `community_reply` | 社区回复表 |
| `post_like` | 帖子点赞关系表 |
| `favorite` | 帖子收藏关系表 |

关键设计：

- `mood_record.ai_reply` 保存心情最新 AI 回复。
- `community_post.ai_reply` 保存帖子最新 AI 回复。
- `ai_reply.target_type + target_id` 关联不同业务场景。
- 帖子和回复通过 `status` 支持隐藏、恢复等管理操作。
- 点赞和收藏使用关系表避免重复操作。

## 10. 安全设计

### 10.1 密码安全

用户密码使用 BCrypt 加密存储，登录时使用 BCrypt 校验，不保存明文密码。

### 10.2 登录认证

系统使用 JWT 保存登录态。前端保存 token，后续请求由请求拦截器自动携带。

### 10.3 权限控制

普通用户只能访问用户端接口。管理员接口校验 `role=ADMIN`。用户私有数据以 token 中解析出的 userId 为准，不信任前端传入的 userId。

### 10.4 AI 失败兜底

AI 服务属于外部依赖，调用失败不能影响心情记录或帖子发布。系统通过兜底文案和失败日志保证主流程稳定。

## 11. 部署概要

本地开发：

```text
前端：http://localhost:5173
后端：http://localhost:8080
数据库：localhost:3306/xinyu
```

生产部署建议：

```text
用户浏览器
  -> Nginx
     -> Vue dist 静态资源
     -> /api 反向代理到 Spring Boot
  -> MySQL
```

后续可以根据需要加入 Redis 缓存热门内容、统计数据或限流数据。

## 12. 概要设计结论

心屿系统当前已经形成“用户记录情绪 -> AI 回复 -> 社区互动 -> 后台治理 -> AI 记录审查 -> 平台统计”的完整业务闭环。

从工程角度看，项目覆盖了前后端分离、JWT 认证、角色权限、MyBatis-Plus 分页筛选、AI 接入、失败兜底、后台管理、组件复用和文档维护等常见实习项目能力。后续增强重点应放在部署、图表、举报审核、Redis 缓存和 AI 异步化上。
