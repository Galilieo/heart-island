# 接口设计说明书

## 1. 文档目的

本文档用于说明“心屿 AI 情绪记录与匿名互助社区系统”的后端 REST 接口设计，包括统一返回结构、认证方式、用户端接口、社区接口、后台管理接口和 AI 回复记录接口。

接口以当前项目实现为准，前端通过 `xinyu-vue/src/api` 目录进行封装，后端接口由 `xinyu-backend` 中的 Controller 提供。

## 2. 通用约定

### 2.1 基础地址

本地开发环境：

```text
http://localhost:8080
```

前端开发环境：

```text
http://localhost:5173
```

### 2.2 认证方式

登录成功后，后端返回 token。前端保存 token，并在后续请求头中携带：

```http
Authorization: Bearer <token>
```

普通用户接口要求已登录。

管理员接口要求：

- 已登录
- 用户角色为 `ADMIN`

### 2.3 统一返回结构

后端接口统一使用 `Result` 返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

常见语义：

| 字段 | 说明 |
| --- | --- |
| `code` | 业务状态码 |
| `message` | 提示信息 |
| `data` | 返回数据 |

前端 api 层会对响应进行统一处理，页面通常直接使用业务数据。

## 3. 用户接口

| 功能 | 方法 | 地址 | 权限 |
| --- | --- | --- | --- |
| 注册 | POST | `/user/register` | 无 |
| 登录 | POST | `/user/login` | 无 |
| 修改个人资料 | POST | `/user/update-profile` | 登录 |
| 修改密码 | POST | `/user/update-password` | 登录 |
| 个人资料统计 | GET | `/user/profile-stats` | 登录 |

### 3.1 注册

```http
POST /user/register
```

请求示例：

```json
{
  "username": "test",
  "password": "123456",
  "nickname": "测试用户"
}
```

说明：

- 用户名不能重复。
- 密码后端使用 BCrypt 加密保存。
- 接口不会返回明文密码。

### 3.2 登录

```http
POST /user/login
```

请求示例：

```json
{
  "username": "test",
  "password": "123456"
}
```

返回 `LoginVO` 含 token 与 `UserVO`。`UserVO` 字段：
`id / username / nickname / gender / birthday / bio / city / role / status`，不含 password。
老用户的 `gender / birthday / bio / city` 可能为 `null`。

### 3.3 修改个人资料

```http
POST /user/update-profile
```

请求示例：

```json
{
  "nickname": "新昵称",
  "gender": "MALE",
  "birthday": "2004-08-12",
  "bio": "一句签名",
  "city": "上海"
}
```

字段约束：

- `nickname`：必填，不超过 50 字。
- `gender`：可选，建议取值 `MALE / FEMALE / OTHER / UNDISCLOSED`，长度不超过 20。
- `birthday`：可选，ISO 日期字符串 `yyyy-MM-dd`；年龄由前端推算，不入库。
- `bio`：可选，最多 300 字。
- `city`：可选，最多 50 字。
- 任何字段传空字符串视为清空（写 NULL）。

响应返回更新后的完整 `UserVO`，前端 store 同步覆盖本地登录用户信息。

### 3.4 修改密码

```http
POST /user/update-password
```

请求示例：

```json
{
  "oldPassword": "123456",
  "newPassword": "654321"
}
```

后端先用 BCrypt 校验原密码，新密码不少于 6 位。

### 3.5 个人资料统计

```http
GET /user/profile-stats
```

返回当前登录用户的统计数字，供个人资料页顶部小条展示：

```json
{
  "moodCount": 12,
  "postCount": 5,
  "favoriteCount": 8,
  "joinDays": 34
}
```

字段说明：

- `moodCount`：当前用户 status=1 的心情记录数。
- `postCount`：当前用户 status=1 的帖子数。
- `favoriteCount`：当前用户收藏的帖子数。
- `joinDays`：从注册当天起累计的天数（含今天）。

## 4. 心情记录接口

| 功能 | 方法 | 地址 | 说明 |
| --- | --- | --- | --- |
| 心情列表 | GET | `/mood/list` | 分页、筛选当前用户心情 |
| 新增心情 | POST | `/mood/add` | 新增后生成 AI 回复 |
| 心情详情 | GET | `/mood/detail/{id}` | 查看当前用户记录详情 |
| 修改心情 | PUT | `/mood/update/{id}` | 修改后重新生成 AI 回复 |
| 删除心情 | DELETE | `/mood/delete/{id}` | 删除当前用户记录 |
| 最近 7 天趋势 | GET | `/mood/trend/recent7` | 统计近期记录 |

### 4.1 心情列表

```http
GET /mood/list?pageNum=1&pageSize=10&moodType=焦虑&keyword=学习
```

返回的记录包含：

- id
- moodType
- moodScore
- content
- aiReply
- createTime
- updateTime

### 4.2 新增心情

```http
POST /mood/add
```

请求示例：

```json
{
  "moodType": "焦虑",
  "moodScore": 6,
  "content": "今天学习压力有点大，但还是想坚持。"
}
```

业务规则：

1. 保存心情记录。
2. 调用 AI 生成陪伴回复。
3. 将最新 AI 回复写入 `mood_record.ai_reply`。
4. 将完整调用过程写入 `ai_reply` 表，`targetType=MOOD_RECORD`。
5. 如果 AI 调用失败，心情记录仍然保存成功，并写入兜底回复和失败日志。

### 4.3 修改心情

```http
PUT /mood/update/{id}
```

修改后会重新生成 AI 回复，并保存新的 AI 调用日志。

### 5.2 帖子接口

| 功能 | 方法 | 地址 | 说明 |
| --- | --- | --- | --- |
| 帖子列表 | GET | `/community/post/list` | 支持话题、我的帖子、排序 |
| 帖子详情 | GET | `/community/post/detail?id=` | 返回帖子、AI 回复和互动状态 |
| 发布帖子 | POST | `/community/post/add` | 发布后生成帖子 AI 回复 |
| 修改帖子 | POST | `/community/post/update` | 修改本人帖子并重新生成 AI 回复 |
| 删除帖子 | POST | `/community/post/delete?id=` | 删除本人帖子 |
| 收藏帖子 | GET | `/community/post/favorites` | 当前用户收藏帖子 |

帖子返回字段包含：

- id
- userId
- topicId
- moodType
- content
- aiReply
- replyCount
- likeCount
- favoriteCount
- liked
- favorited
- status
- createTime

发布帖子业务规则：

1. 保存帖子内容。
2. 根据话题、情绪和正文生成 AI 心屿回复。
3. 最新回复写入 `community_post.ai_reply`。
4. 调用日志写入 `ai_reply` 表，`targetType=POST`。
5. AI 失败不影响帖子发布。

## 11. 后台 AI 回复记录接口

```http
GET /admin/ai-replies/page
```

权限：管理员。

查询参数：

| 参数 | 说明 |
| --- | --- |
| `pageNum` | 页码 |
| `pageSize` | 每页数量 |
| `userId` | 用户 id |
| `targetType` | 业务类型，如 `MOOD_RECORD`、`POST` |
| `status` | 调用状态，成功或失败 |
| `keyword` | prompt 或回复内容关键词 |

返回字段：

| 字段 | 说明 |
| --- | --- |
| `id` | AI 记录 id |
| `userId` | 用户 id |
| `modelName` | 模型名称 |
| `prompt` | 提示词内容 |
| `replyContent` | AI 回复内容 |
| `targetType` | 业务类型 |
| `targetId` | 业务数据 id |
| `status` | 调用状态 |
| `errorMessage` | 错误信息 |
| `createTime` | 创建时间 |

说明：

- 心情记录 AI 回复使用 `targetType=MOOD_RECORD`。
- 社区帖子 AI 回复使用 `targetType=POST`。
- 即使 AI 调用失败，也会保留失败记录，便于管理员排查。

## 12. 前端接口封装约定

前端接口按业务域拆分：

```text
xinyu-vue/src/api
├─ user.js
├─ mood.js
├─ community.js
├─ admin.js
├─ index.js
└─ _helpers.js
```

页面不直接写 axios 请求，而是调用 api 层方法，再由 Pinia store 或页面逻辑管理状态。

这种拆分的好处：

- 请求路径集中维护
- 页面代码更干净
- 后端接口变动时修改范围更小
- 便于后台管理模块复用筛选和分页逻辑

## 13. 接口设计结论

当前接口已经覆盖用户端、社区端、管理端和 AI 记录管理。系统接口不是简单 CRUD，而是包含认证、权限、分页筛选、状态管理、AI 失败兜底和日志留痕等真实 Web 项目中常见的业务细节。
