# 心屿 AI 情绪记录与匿名互助社区系统

# 06-详细设计说明书

## 1. 文档目的

本文档用于说明心屿 AI 情绪记录与匿名互助社区系统各核心模块的详细设计方案，包括模块职责、输入输出、处理流程、关键业务规则、异常处理和扩展点。

详细设计承接需求分析、概要设计、数据库设计和接口设计，重点回答以下问题：

1. 每个模块内部如何处理业务。
2. 前端请求到后端响应之间的数据如何流转。
3. 登录认证、密码加密、权限校验如何实现。
4. 心情记录、AI 回复、社区互动和后台审核如何落地。
5. 哪些逻辑需要抽象为可复用模块。

## 2. 系统分层设计

### 2.1 后端分层

后端采用 Controller、Service、Mapper、Entity、DTO、VO、Utils、Config、Exception 的分层结构。

| 层级 | 职责 |
| --- | --- |
| Controller | 接收请求、基础参数校验、调用 Service、返回 Result |
| Service | 处理业务逻辑和事务边界 |
| Mapper | 访问数据库 |
| Entity | 映射数据库表 |
| DTO | 接收前端请求参数 |
| VO | 返回前端展示数据 |
| Utils | 通用工具，如 JWT 工具 |
| Config | 配置类，如 CORS、密码加密器 |
| Exception | 全局异常处理 |

### 2.2 前端分层

前端采用 Vue 页面、路由、请求封装和本地状态存储的结构。

| 层级 | 职责 |
| --- | --- |
| views | 页面级组件，如登录页、首页、社区页 |
| router | 路由跳转与页面访问控制 |
| utils/request.js | axios 封装，统一 baseURL 和 token |
| localStorage | 保存 loginUser 和 token |
| components | 可复用 UI 组件 |

## 3. 统一返回模块详细设计

### 3.1 模块职责

统一返回模块负责规范后端所有接口响应结构，使前端能够用统一方式处理成功和失败结果。

### 3.2 返回结构

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

### 3.3 设计规则

1. 成功时 code 为 200。
2. 业务失败或异常时 code 为 500。
3. message 用于给前端展示提示。
4. data 用于返回具体业务数据。
5. Controller 不直接返回 Entity 或字符串，统一返回 Result。

### 3.4 处理流程

```text
Controller 处理请求
-> 调用 Service
-> 得到业务结果
-> Result.success 或 Result.error
-> 前端根据 code 判断成功或失败
```

## 4. 全局异常处理详细设计

### 4.1 模块职责

全局异常处理模块负责捕获系统中未手动处理的异常，避免异常堆栈直接暴露给前端。

### 4.2 处理流程

```text
业务代码抛出异常
-> GlobalExceptionHandler 捕获异常
-> 后端打印异常日志
-> 返回统一错误 Result
```

### 4.3 返回示例

```json
{
  "code": 500,
  "message": "服务异常，请稍后再试",
  "data": null
}
```

### 4.4 设计说明

全局异常处理适合处理系统未知异常。对于可预期的业务错误，如用户名重复、密码错误、参数为空，建议在业务代码中主动返回明确提示。

## 5. 用户注册详细设计

### 5.1 功能说明

用户通过用户名、密码和昵称注册账号。注册成功后，系统保存加密后的密码。

### 5.2 输入数据

| 字段 | 说明 |
| --- | --- |
| username | 用户名 |
| password | 明文密码 |
| nickname | 昵称 |

### 5.3 处理流程

```text
前端提交注册表单
-> Controller 校验 username、password、nickname 是否为空
-> Service 根据 username 查询用户是否存在
-> 如果已存在，返回用户名已存在
-> 如果不存在，使用 BCrypt 加密密码
-> 保存用户到 user 表
-> 返回注册成功
```

### 5.4 关键规则

1. username 不能为空。
2. password 不能为空。
3. nickname 不能为空。
4. username 不能重复。
5. password 不能明文保存。
6. 默认 role 为 USER。
7. 默认 status 为正常。

### 5.5 异常情况

| 情况 | 返回提示 |
| --- | --- |
| 用户名为空 | 用户名不能为空 |
| 密码为空 | 密码不能为空 |
| 昵称为空 | 昵称不能为空 |
| 用户名重复 | 用户名已存在 |

## 6. 用户登录与个人资料详细设计

### 6.1 功能说明

用户输入用户名和密码进行登录。登录成功后，系统返回用户基本信息和 JWT token。

### 6.2 处理流程

```text
前端提交用户名和密码
-> Controller 校验参数是否为空
-> Service 根据 username 查询用户
-> 用户不存在，返回账号或密码错误
-> 用户存在，使用 BCrypt matches 校验密码
-> 密码错误，返回账号或密码错误
-> 校验用户状态是否正常
-> 生成 JWT token
-> 组装 UserVO
-> 组装 LoginVO
-> 返回登录成功
-> 前端保存 loginUser 和 token
```

### 6.3 密码校验设计

注册时：

```text
明文密码 -> BCrypt 加密 -> 保存密文
```

登录时：

```text
用户输入明文密码 + 数据库密文密码
-> BCrypt matches
-> 返回是否匹配
```

### 6.4 返回对象设计

登录成功后返回 LoginVO：

```json
{
  "user": {
    "id": 1,
    "username": "test",
    "nickname": "小屿",
    "avatar": null
  },
  "token": "xxxxx.yyyyy.zzzzz"
}
```

### 6.5 设计说明

后端不能直接返回 User 实体，因为 User 中包含 password 字段。应使用 UserVO 控制返回字段。

### 6.6 修改昵称

```text
用户在个人资料页提交新昵称
-> 后端校验 token 并解析 userId
-> 校验昵称非空且不超过 50 字
-> 更新当前用户的昵称
-> 返回更新后的用户信息（不含 password）
-> 前端同步更新本地登录用户信息
```

设计说明：从 token 解析 userId，只允许修改当前登录用户；用户名不可修改。

### 6.7 修改密码

```text
用户在个人资料页提交原密码和新密码
-> 后端校验 token 并解析 userId
-> 使用 BCrypt matches 校验原密码
-> 原密码不正确，返回错误提示
-> 原密码正确，将新密码 BCrypt 加密后保存
-> 返回修改成功
```

设计说明：修改密码必须先验证原密码，避免 token 被盗用后直接改密；新密码同样以密文存储。

## 7. JWT 认证详细设计

### 7.1 模块职责

JWT 模块负责生成、解析和校验 token，用于识别当前请求用户身份。

### 7.2 token 内容

token 中建议包含：

1. userId。
2. username。
3. expireTime。

不应包含：

1. password。
2. 手机号等敏感信息。
3. 大量业务数据。

### 7.3 生成流程

```text
用户登录成功
-> 获取 loginUser.id 和 loginUser.username
-> JwtUtil.generateToken(userId, username)
-> 返回 token 给前端
```

### 7.4 携带流程

```text
前端 localStorage 保存 token
-> request.js 请求拦截器读取 token
-> 设置请求头 token
-> 后端接口通过 @RequestHeader 获取 token
```

### 7.5 解析流程

```text
后端获取请求头 token
-> JwtUtil 校验签名
-> 解析 userId
-> 根据 userId 处理当前用户数据
```

### 7.6 设计原则

1. 后端以 token 解析出的 userId 作为身份依据。
2. 后端不信任前端传入的 userId。
3. token 过期后应提示用户重新登录。
4. 管理员接口还需要基于 userId 查询 role。

## 8. 心情记录模块详细设计

## 8.1 新增心情记录

### 8.1.1 功能说明

用户新增一条心情记录，系统自动绑定当前登录用户。

### 8.1.2 处理流程

```text
前端提交 moodType、moodScore、content、isPublic
-> request.js 自动携带 token
-> Controller 校验 token 是否存在
-> Controller 校验 moodType 和 content
-> JwtUtil 从 token 解析 userId
-> 后端将 userId 设置到 MoodRecord
-> Service 保存记录
-> 可触发 AI 回复生成
-> 返回新增成功
```

### 8.1.3 关键规则

1. 必须登录。
2. 心情类型不能为空。
3. 心情内容不能为空。
4. userId 由后端从 token 中解析。
5. create_time 由数据库或后端自动生成。

## 8.2 查询心情记录列表

### 8.2.1 功能说明

用户查询自己的心情记录列表。

### 8.2.2 处理流程

```text
前端请求心情列表
-> request.js 自动携带 token
-> 后端解析 token 获取 userId
-> Service 根据 userId 查询 mood_record
-> 按 create_time 倒序排序
-> 返回列表
```

### 8.2.3 查询条件

后续可支持：

1. 心情类型。
2. 开始日期。
3. 结束日期。
4. 当前页码。
5. 每页数量。

### 8.2.4 设计说明

旧版本可能通过前端传 userId 查询记录，但正式设计应从 token 中解析 userId，避免用户篡改参数访问他人记录。

## 8.3 查看心情记录详情

### 8.3.1 处理流程

```text
前端传入记录 id
-> 后端解析 token 获取 userId
-> 根据 id 查询记录
-> 判断记录是否存在
-> 判断记录 user_id 是否等于当前 userId
-> 返回记录详情
```

### 8.3.2 异常情况

| 情况 | 返回提示 |
| --- | --- |
| 记录不存在 | 记录不存在 |
| 记录不属于当前用户 | 无权限访问 |

## 8.4 修改心情记录

### 8.4.1 处理流程

```text
前端传入记录 id 和修改内容
-> 后端解析 token 获取 userId
-> 查询记录是否存在
-> 校验记录所属用户
-> 校验参数
-> 更新记录
-> 返回修改成功
```

## 8.5 删除心情记录

### 8.5.1 处理流程

```text
前端传入记录 id
-> 后端解析 token 获取 userId
-> 查询记录是否存在
-> 校验记录所属用户
-> 删除记录或逻辑删除
-> 返回删除成功
```

### 8.5.2 设计说明

当前最小闭环阶段可以先按 id 删除。正式版本建议增加所属用户校验，避免删除他人记录。

## 9. AI 回复模块详细设计

### 9.1 模块职责

AI 回复模块负责根据用户输入内容生成温暖回复，并保存 AI 调用记录。

### 9.2 心情记录 AI 回复流程

```text
用户新增心情记录
-> 后端保存 mood_record
-> 后端组装 prompt
-> 调用大模型 API
-> 获取 AI 回复
-> 保存 ai_reply 记录
-> 可回写 mood_record.ai_reply
-> 返回 AI 回复给前端
```

### 9.3 prompt 设计原则

AI 回复应遵循以下原则：

1. 语气温和。
2. 不做医疗诊断。
3. 不承诺绝对结果。
4. 不替代专业心理咨询。
5. 可以给出简单陪伴、鼓励和可执行的小建议。

### 9.4 AI 回复记录设计

ai_reply 表记录：

1. user_id。
2. target_type。
3. target_id。
4. prompt。
5. reply_content。
6. model_name。
7. status。
8. error_message。
9. create_time。

### 9.5 异常处理

AI 调用失败时：

1. 心情记录保存不应回滚。
2. 系统可以保存失败记录。
3. 前端提示“AI 回复暂时生成失败，请稍后再试”。
4. 后续可增加重试机制。

## 10. 匿名社区模块详细设计

## 10.1 发布帖子

### 10.1.1 处理流程

```text
用户填写 topicId、moodType、content
-> 前端提交帖子
-> 后端校验 token 并解析 userId
-> 校验话题、情绪非空，内容非空且不超过 800 字
-> 保存帖子，匿名昵称默认「匿名岛民」，status 置 1
-> 根据话题名 + 心情 + 内容调用 AiReplyService.generateReplyForPost
-> 将 AI 回复回写到 community_post.ai_reply，并向 ai_reply 表写一行 target_type=POST 的日志
-> AI 调用失败时使用默认陪伴文案兜底，不阻断发帖
-> 返回发布成功
```

### 10.1.2 状态设计

| status | 说明 |
| --- | --- |
| 0 | 已删除 |
| 1 | 正常 |

说明：当前帖子采用逻辑删除（status 在 1 与 0 之间切换）。审核相关的待审核、已拒绝状态属于后台管理阶段，本阶段未启用。

### 10.1.3 匿名设计

帖子表保存真实 user_id，方便用户管理自己的帖子和后续管理员治理；对外展示统一使用匿名昵称字段 anonymous_name（默认「匿名岛民」），不展示真实昵称。

## 10.2 帖子列表

### 10.2.1 处理流程

```text
前端请求帖子列表
-> 传入 topicId（可选）、mine（可选）、sort（可选）
-> 后端只查询 status 为正常的帖子
-> topicId 存在则按话题过滤；mine 为 true 则只查当前用户的帖子
-> sort=hot 按点赞数倒序（创建时间兜底），否则按创建时间倒序
-> 携带 token 时为每条帖子标记当前用户的点赞、收藏状态
-> 返回帖子列表
```

### 10.2.2 收藏帖子列表

```text
前端请求 /community/post/favorites
-> 后端校验 token 并解析 userId
-> 查询 favorite 表中该用户的收藏记录，按收藏时间倒序
-> 逐条取出对应帖子，跳过已删除或隐藏的帖子
-> 返回收藏的帖子列表
```

## 10.3 回复帖子

### 10.3.1 处理流程

```text
用户提交 postId 和 content
-> 后端校验 token 并解析 userId
-> 校验帖子存在且正常，回复内容非空且不超过 500 字
-> 保存回复
-> 返回回复成功
```

说明：回复支持作者本人修改和删除；修改、删除均校验回复归属当前用户。

## 10.4 点赞帖子

### 10.4.1 处理流程

```text
用户点击点赞
-> 后端校验 token
-> 查询 post_like 是否已存在
-> 如果已存在，返回已点赞
-> 如果不存在，新增点赞记录
-> 更新帖子 like_count
-> 返回点赞成功
```

取消点赞则删除 post_like 记录并将 like_count 减一。

### 10.4.2 设计说明

post_like 表使用 post_id + user_id 唯一约束，防止重复点赞。

## 10.5 收藏帖子

### 10.5.1 处理流程

```text
用户点击收藏
-> 后端校验 token
-> 查询 favorite 是否已存在
-> 如果已存在，返回已收藏
-> 如果不存在，新增收藏记录
-> 返回收藏成功
```

取消收藏则删除 favorite 记录。收藏的帖子可在个人收藏页查看（见 10.2.2）。

### 10.5.2 设计说明

favorite 表使用 post_id + user_id 唯一约束，防止重复收藏。

## 10.6 帖子 AI 回复

### 10.6.1 触发时机

发帖（10.1）和修改帖子时同步生成；老帖（功能上线前发布的）不批量回填。

### 10.6.2 数据落点

1. 最新一次成功回复（或失败兜底文案）冗余到 `community_post.ai_reply`。
2. 每次调用（无论成功失败）都向 `ai_reply` 表写一行，target_type 固定为 `POST`，target_id 为帖子 id。

### 10.6.3 失败处理

`AiReplyService.generateReplyForPost` 内部捕获异常返回默认陪伴文案，外层只读 `replyContent` 写库，不抛出。AI 调用不进事务。

## 11. 后台管理模块详细设计

## 11.1 管理员权限校验

### 11.1.1 处理流程

```text
管理员接口收到请求
-> 读取 token
-> 解析 userId
-> 查询用户 role
-> role 不是 ADMIN，返回权限不足
-> role 是 ADMIN，继续执行业务
```

## 11.2 用户管理

管理员可以查询用户列表、搜索用户、禁用或启用用户。

处理流程：

```text
管理员请求用户列表
-> 校验管理员权限
-> 按用户名、状态分页查询
-> 返回用户列表
```

禁用用户流程：

```text
管理员选择用户
-> 修改 user.status
-> 禁用后用户不能发布内容或正常使用受保护功能
```

## 11.3 帖子审核

处理流程：

```text
管理员查看待审核帖子
-> 查看帖子详情
-> 选择通过或拒绝
-> 更新 post.status
-> 返回审核结果
```

## 11.4 评论管理

处理流程：

```text
管理员查看评论列表
-> 发现违规评论
-> 删除评论或修改 status
-> 更新帖子评论数
-> 返回处理成功
```

## 11.5 话题管理

管理员可以新增、修改、启用、禁用话题。

设计说明：

话题禁用后，普通用户不能在该话题下发布新帖子，但历史帖子可以继续按业务策略展示或隐藏。

## 12. 数据统计模块详细设计

## 12.1 用户端情绪统计

### 12.1.1 最近 7 天心情趋势

处理流程：

```text
后端解析 token 获取 userId
-> 查询最近 7 天 mood_record
-> 按日期分组
-> 计算平均 mood_score 或记录数量
-> 返回前端用于图表展示
```

### 12.1.2 心情类型占比

处理流程：

```text
后端解析 token 获取 userId
-> 查询用户 mood_record
-> 按 mood_type 分组统计数量
-> 返回各类型数量和占比
```

## 12.2 管理端统计

平台概览统计包括：

1. 用户总数。
2. 今日新增用户。
3. 帖子总数。
4. 今日新增帖子。
5. 评论总数。
6. AI 调用次数。

处理方式：

1. 使用 SQL count 和 group by 聚合查询。
2. 高频统计可缓存到 Redis。
3. 定时刷新或按需刷新统计结果。

## 13. 前端请求封装详细设计

### 13.1 request.js 职责

request.js 负责封装 axios，避免每个页面重复写后端地址和 token 请求头。

### 13.2 请求流程

```text
页面调用 request.get/post/put/delete
-> request.js 自动读取 localStorage token
-> token 存在则加入请求头
-> 发送请求到后端
-> 后端处理并返回 Result
-> 页面根据 res.data.code 处理结果
```

### 13.3 后续扩展

request.js 后续可增加响应拦截器：

1. token 过期自动跳转登录页。
2. 统一处理 code 非 200 的提示。
3. 统一处理网络错误。

## 14. 路由访问控制详细设计

### 14.1 用户端路由控制

首页、社区列表页、帖子详情页、个人收藏页、个人资料页等需要登录后访问。

处理方式：

1. 在路由 meta 上标记 requiresAuth。
2. 通过 router.beforeEach 全局守卫统一拦截：访问带 requiresAuth 的页面时检查 token，不存在则跳转登录页。
3. 已登录用户访问登录页或注册页时，自动跳转首页。

当前已配置的用户端路由：登录页 `/`、注册页 `/register`、首页 `/home`、社区列表页 `/posts`、帖子详情页 `/community/post/:id`、个人收藏页 `/favorites`、个人资料页 `/profile`。

### 14.2 管理端路由控制

管理端页面需要管理员登录后访问。

处理方式：

1. 检查 token 是否存在。
2. 检查当前用户 role 是否为 ADMIN。
3. 非管理员跳转无权限页面或登录页。

## 15. 数据安全详细设计

### 15.1 不返回 password

用户登录或查询用户信息时，后端返回 UserVO，不直接返回 User 实体。

### 15.2 不信任前端 userId

与用户数据相关的接口，后端都从 token 中解析 userId。即使前端传入 userId，后端也不能直接作为身份依据。

### 15.3 用户数据隔离

心情记录、收藏、个人信息等用户私有数据都需要按当前 userId 过滤。

### 15.4 社区匿名保护

匿名帖子前端不展示真实用户名，但数据库保留真实 user_id，用于内容治理和用户本人管理。

## 17. 详细设计结论

本系统详细设计围绕用户认证、情绪记录、AI 回复、匿名社区、后台管理和数据统计展开。系统通过统一 Result、全局异常处理、JWT 认证、BCrypt 密码加密、VO 返回对象和后端解析 userId 等设计保证接口规范性和数据安全性。

业务模块之间保持相对独立：用户模块提供身份基础，心情记录模块提供私人记录能力，AI 回复模块提供智能回复能力，社区模块提供公开互动能力，后台模块提供内容治理能力，统计模块提供数据分析能力。该设计能够支撑系统从最小闭环逐步扩展为完整的平台型 Web 应用。
