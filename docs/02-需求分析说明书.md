# 心屿 AI 情绪记录与匿名互助社区系统

# 02-需求分析说明书

## 1. 文档目的

本文档用于明确“心屿 AI 情绪记录与匿名互助社区系统”的产品需求、用户角色、功能范围、业务规则、非功能需求、测试关注点和阶段边界。

本文档采用“软件工程需求分析 + PRD 产品需求文档”的写法，用于说明系统要解决的问题、面向的用户、功能范围、业务规则和后续迭代边界。

## 2. 问题陈述

### 2.1 用户问题

许多学生和年轻用户在学习、工作、社交和生活中会产生压力、焦虑、低落或迷茫等情绪。他们往往需要一个低压力、低门槛的空间来记录和表达情绪，但传统日记应用只有记录功能，缺少及时反馈；普通社区平台虽然有互动，但用户可能担心身份暴露、被评价或内容不够私密。

因此，用户需要一个既能私人记录情绪，又能获得温暖反馈，还能在匿名社区中安全表达的系统。

### 2.2 项目问题

从系统完整性角度看，普通“登录注册 + 单表 CRUD”功能业务复杂度较低，难以支撑完整的平台型应用。心屿项目需要在最小闭环基础上继续扩展，形成包含用户体系、情绪记录、AI 回复、匿名社区、后台审核和数据统计的完整 Web 系统。

该系统需要能体现：

1. 用户认证和权限控制。
2. 密码加密和敏感字段保护。
3. 多表业务建模。
4. 社区互动关系设计。
5. AI 能力接入与结果持久化。
6. 后台管理和内容治理。
7. 前后端分离和接口规范。
8. 数据统计和可视化展示。

## 3. 解决方案

系统设计为“AI 情绪记录 + 匿名互助社区 + 后台管理”的综合平台。

普通用户可以注册登录、记录心情、查看历史记录、获得 AI 温暖回复，也可以匿名发布帖子、评论、点赞、收藏和浏览话题内容。管理员可以管理用户、审核帖子、删除违规评论、维护话题和查看统计数据。AI 助手作为系统能力，为心情记录、社区帖子和情绪总结提供智能回复。

系统以 Spring Boot + MyBatis-Plus + MySQL 为后端核心，以 Vue3 + axios 为前端核心，以 JWT 维护登录态，以 BCrypt 保护密码安全，后续可引入 Redis、ECharts 和大模型 API 增强项目亮点。

## 4. 系统目标

### 4.1 产品目标

1. 为用户提供安全、温和、低压力的情绪记录空间。
2. 通过 AI 回复提升情绪记录的陪伴感。
3. 通过匿名社区提供情绪表达和互助互动能力。
4. 通过后台管理保障社区内容安全。
5. 通过数据统计帮助用户观察情绪变化。

### 4.2 技术目标

1. 完成前后端分离项目开发。
2. 使用 JWT 实现登录认证。
3. 使用 BCrypt 实现密码加密。
4. 使用 MyBatis-Plus 实现分页、条件查询和基础 CRUD。
5. 使用统一 Result 和全局异常处理规范接口返回。
6. 使用 request.js 封装 axios 并自动携带 token。
7. 使用数据库表关系支撑帖子、评论、点赞、收藏等业务。
8. 后续使用 Redis、ECharts、大模型 API 提升项目技术含量。

### 4.3 工程建设目标

1. 系统不止停留在基础 CRUD，而是具备完整业务场景。
2. 系统能够体现认证、权限、加密、AI、社区、后台、统计等工程能力。
3. 系统模块划分、数据关系、接口设计和安全设计应保持清晰。
4. 系统可分阶段开发，每个阶段都有可运行、可验证的成果。

## 7. 功能需求

## 7.1 用户与权限模块

### 7.1.1 用户注册

用户可以通过用户名、密码和昵称注册账号。

处理规则：

1. 用户名不能为空。
2. 密码不能为空。
3. 昵称不能为空。
4. 用户名不能重复。
5. 密码保存到数据库前必须使用 BCrypt 加密。
6. 注册成功后返回统一 Result。

### 7.1.2 用户登录

用户输入用户名和密码进行登录。

处理规则：

1. 用户名不能为空。
2. 密码不能为空。
3. 后端根据用户名查询用户信息。
4. 使用 BCrypt 校验用户输入密码和数据库加密密码是否匹配。
5. 登录成功后生成 JWT token。
6. 返回用户基本信息和 token。
7. 返回用户信息时不能包含 password 字段。

### 7.1.3 退出登录

用户点击退出登录后，前端清除 localStorage 中的 loginUser 和 token，并跳转到登录页。

### 7.1.4 当前用户信息

登录用户可以查看自己的基本信息。

处理规则：

1. 请求必须携带 token。
2. 后端从 token 中解析 userId。
3. 根据 userId 查询用户信息。
4. 返回 UserVO，不返回 password。

### 7.1.5 权限控制

系统区分普通用户和管理员。

处理规则：

1. 用户表使用 role 字段区分 USER 和 ADMIN。
2. 普通接口要求用户登录。
3. 管理员接口要求登录用户角色为 ADMIN。
4. 未登录访问受保护接口时返回“请先登录”。
5. 无权限访问管理员接口时返回“权限不足”。

## 7.2 情绪记录模块

### 7.2.1 新增心情记录

用户可以记录当前心情类型、心情分数、心情内容和是否公开。

处理规则：

1. 请求必须携带 token。
2. 后端从 token 中解析 userId。
3. 后端不信任前端传入的 userId。
4. 心情类型不能为空。
5. 心情内容不能为空。
6. 心情分数应在 1 到 10 范围内。
7. 保存记录时自动绑定当前登录用户。
8. 新增成功后可以触发 AI 回复生成。

### 7.2.2 查看心情记录列表

用户可以查看自己的历史心情记录。

处理规则：

1. 请求必须携带 token。
2. 后端从 token 中解析 userId。
3. 只查询当前登录用户的心情记录。
4. 默认按创建时间倒序排序。
5. 支持分页查询。
6. 支持按日期和心情类型筛选。

### 7.2.3 查看心情记录详情

用户可以查看某条心情记录的完整内容，包括心情类型、心情分数、心情内容、AI 回复和创建时间。

处理规则：

1. 请求必须携带 token。
2. 用户只能查看自己的心情记录。
3. 如果记录不存在，返回“记录不存在”。
4. 如果记录不属于当前用户，返回“无权限访问”。

### 7.2.4 修改心情记录

用户可以修改自己的心情记录。

处理规则：

1. 请求必须携带 token。
2. 用户只能修改自己的记录。
3. 心情类型、心情内容和心情分数需要进行基础校验。
4. 修改后更新 update_time。

### 7.2.5 删除心情记录

用户可以删除自己的心情记录。

处理规则：

1. 请求必须携带 token。
2. 后端校验记录是否存在。
3. 后端校验记录是否属于当前用户。
4. 校验通过后删除记录。

## 7.3 AI 回复模块

### 7.3.1 心情记录 AI 回复

用户新增心情记录后，系统根据心情内容生成温暖、支持性的 AI 回复。

处理规则：

1. 用户必须已登录。
2. 系统保存心情记录后调用 AI 接口。
3. AI 输入内容应包含心情类型、心情分数和心情内容。
4. AI 回复应避免诊断式、绝对化和过度承诺表达。
5. AI 回复保存到 ai_reply 表，也可回写到 mood_record 的 ai_reply 字段。
6. AI 调用失败时，不影响心情记录保存。

### 7.3.2 帖子 AI 回复

用户发布社区帖子后，系统可以根据帖子内容生成一条温暖评论或建议。

处理规则：

1. 用户必须已登录。
2. 帖子内容不能为空。
3. AI 回复与帖子通过 target_type 和 target_id 关联。
4. 管理员可以查看 AI 回复记录。

### 7.3.3 AI 回复历史

系统保存 AI 回复记录，便于用户回顾，也便于管理员审查。

记录内容：

1. 用户 id。
2. 关联业务类型。
3. 关联业务 id。
4. 用户输入内容。
5. AI 回复内容。
6. 模型名称。
7. 创建时间。

## 7.4 匿名社区模块

### 7.4.1 发布帖子

用户可以在社区发布帖子，表达情绪、分享经历或寻求支持。

处理规则：

1. 用户必须登录。
2. 标题不能为空。
3. 内容不能为空。
4. 话题必须存在且启用。
5. 帖子保存时绑定当前用户。
6. 如果选择匿名，前端展示时隐藏真实用户名。
7. 帖子状态默认为正常或待审核。

### 7.4.2 查看帖子列表

用户可以浏览社区帖子列表。

处理规则：

1. 支持分页查询。
2. 支持按话题筛选。
3. 支持按发布时间排序。
4. 支持按热度排序。
5. 只展示状态正常的帖子。

### 7.4.3 查看帖子详情

用户可以查看帖子详情，包括标题、内容、话题、发布时间、点赞数、评论数和评论列表。

### 7.4.4 评论帖子

用户可以对帖子发表评论。

处理规则：

1. 用户必须登录。
2. 评论内容不能为空。
3. 帖子必须存在且状态正常。
4. 评论保存时绑定当前用户。
5. 帖子评论数增加。

### 7.4.5 点赞帖子

用户可以点赞帖子，也可以取消点赞。

处理规则：

1. 用户必须登录。
2. 同一用户对同一帖子只能点赞一次。
3. 点赞时写入 post_like 表。
4. 取消点赞时删除对应记录。
5. 帖子 like_count 对应更新。

### 7.4.6 收藏帖子

用户可以收藏帖子，也可以取消收藏。

处理规则：

1. 用户必须登录。
2. 同一用户对同一帖子只能收藏一次。
3. 收藏时写入 favorite 表。
4. 取消收藏时删除对应记录。

## 7.5 后台管理模块

### 7.5.1 用户管理

管理员可以查看用户列表、搜索用户、按状态筛选用户、禁用用户和启用用户。

### 7.5.2 帖子管理

管理员可以查看帖子列表、按状态筛选帖子、查看帖子详情、审核通过帖子、拒绝帖子和删除违规帖子。

帖子状态建议：

1. 0：待审核。
2. 1：正常。
3. 2：已拒绝。
4. 3：已删除。

### 7.5.3 评论管理

管理员可以查看评论列表、按帖子查询评论、删除评论和按状态筛选评论。

### 7.5.4 话题管理

## 8. 实现决策

### 8.1 模块划分决策

系统按以下模块拆分：

1. 用户与权限模块。
2. 情绪记录模块。
3. AI 回复模块。
4. 匿名社区模块。
5. 后台管理模块。
6. 数据统计模块。

该划分能够让每个模块职责清晰，同时便于分阶段开发。

### 8.2 登录认证决策

系统使用 JWT 维护登录状态。登录成功后后端生成 token，前端保存 token，后续请求自动携带 token。后端从 token 中解析 userId，并基于该 userId 查询用户数据。

该设计避免后端直接信任前端传入的 userId，降低越权访问风险。

### 8.3 密码安全决策

系统使用 BCrypt 加密用户密码。数据库只保存加密后的密码，登录时通过 matches 方法比较用户输入明文密码和数据库密文密码。

### 8.4 用户返回对象决策

登录成功后返回 UserVO，而不是 User 实体。UserVO 只包含 id、username、nickname、avatar 等展示字段，不包含 password。

### 8.5 AI 回复模块决策

AI 回复设计为独立模块，通过 target_type 和 target_id 关联不同业务场景。target_type 可以是 MOOD_RECORD、POST 或 CHAT。

这种设计比把 AI 回复写死在心情记录表中更灵活，后续可以扩展到帖子回复、聊天消息和情绪总结。

### 8.6 社区互动决策

点赞和收藏使用独立关联表记录用户行为，避免只在帖子表中简单维护数字。帖子表可以保留 like_count、comment_count 等冗余字段用于列表展示，但真实用户行为应以 post_like、favorite 表为准。

### 8.7 内容审核决策

帖子和评论使用 status 字段表示内容状态。管理员通过后台修改状态，而不是直接物理删除所有内容。这样便于保留审核记录和后续扩展。

## 10. 非功能需求

### 10.1 安全性需求

1. 用户密码必须加密存储，不能保存明文密码。
2. 登录成功后返回 token，不返回 password。
3. 需要登录的接口必须校验 token。
4. 后端不能直接信任前端传入的 userId。
5. 管理员接口必须校验用户角色。
6. 异常信息不能直接暴露后端堆栈。

### 10.2 易用性需求

1. 用户操作流程应简单清晰。
2. 表单校验失败时应给出明确提示。
3. 登录失效时应提示用户重新登录。
4. 无数据时应展示友好的空状态提示。
5. 页面风格应温和、轻量、低压力。

### 10.3 可维护性需求

1. 后端采用 Controller、Service、Mapper 分层。
2. 返回结果统一使用 Result。
3. 异常统一由 GlobalExceptionHandler 处理。
4. 前端请求统一通过 request.js 封装。
5. 各模块职责清晰，便于后续扩展。

### 10.4 可扩展性需求

1. AI 回复模块应独立设计，便于关联不同业务场景。
2. 社区模块应支持后续扩展点赞、收藏、话题和审核。
3. 统计模块应支持后续接入 ECharts。
4. 热门数据或统计数据可后续接入 Redis 缓存。
5. 文件上传功能可用于用户头像扩展。

### 10.5 性能需求

1. 列表查询应支持分页，避免一次性加载大量数据。
2. 帖子列表、评论列表和用户列表应支持条件查询。
3. 热门话题和统计数据可使用缓存优化。
4. AI 调用耗时较长时，后续可考虑异步处理。

## 14. 需求分析结论

心屿 AI 情绪记录与匿名互助社区系统的需求可以分为用户端、管理端和 AI 能力三部分。用户端以情绪记录和匿名社区互动为核心，管理端以内容治理和数据统计为核心，AI 能力则为项目提供差异化亮点。

从产品价值看，系统解决了用户低压力记录情绪、获得陪伴反馈和匿名表达的问题。从工程价值看，系统覆盖了认证、权限、加密、接口、数据库、社区互动、后台管理、统计和 AI 接入等常见 Web 应用能力。

项目应先完成注册登录、JWT、心情记录管理等最小闭环，再逐步扩展 AI 回复、匿名社区、后台管理和数据统计。这样的需求拆分既能保证项目持续可运行，也有利于后续迭代和维护。
