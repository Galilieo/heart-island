# 10-项目总结与后续规划

## 1. 文档目的

本文档用于对心屿 AI 情绪记录与匿名互助社区系统进行阶段性总结，并说明后续可持续迭代方向。

项目总结与后续规划主要回答以下问题：

## 2. 项目总体总结

心屿 AI 情绪记录与匿名互助社区系统定位为面向个人情绪记录、AI 情绪陪伴和匿名社区表达的 Web 应用。

系统围绕用户的情绪表达场景展开，提供注册登录、心情记录、AI 回复、匿名发帖、评论互动、内容管理和数据统计等能力。整体设计采用前后端分离架构，后端负责用户认证、业务处理、数据持久化和接口输出，前端负责页面展示、用户交互和请求封装。

从系统建设角度看，心屿不是单纯的 CRUD 练习，而是一个围绕真实业务场景拆分模块、设计权限边界、管理数据关系、考虑后续扩展的综合型应用。

## 3. 当前阶段完成情况

### 3.1 已完成的基础能力

当前系统已完成阶段 0 至阶段 3 的能力：

9. 最近 7 天心情趋势统计。

匿名社区：

10. 发布、修改、删除匿名帖子（逻辑删除）。

13. 点赞/取消点赞、收藏/取消收藏、个人收藏页。

工程基础：

14. 统一 Result 返回结构、全局异常处理、CORS 配置。
15. 集成 Knife4j 接口文档。
16. 前后端接口联调，Apifox、浏览器 Network 与数据库验证。


## 4. 产品设计总结

### 4.1 产品定位

心屿系统的核心定位是：

```text
记录情绪
理解情绪
获得回应
建立连接
观察变化
```

系统希望帮助用户将情绪从模糊状态转化为可记录、可回顾、可分析的内容，并通过 AI 回复和匿名社区降低表达门槛。

### 4.2 用户价值

系统面向普通用户提供以下价值：

1. 通过心情记录保存个人情绪轨迹。
2. 通过 AI 回复获得即时、温和的反馈。
3. 通过匿名社区表达不适合公开社交平台发布的内容。
4. 通过评论、点赞、收藏获得轻量互动。
5. 通过情绪统计了解自己一段时间内的情绪变化。

### 4.3 管理价值

系统面向平台管理提供以下价值：

1. 用户状态管理。
2. 帖子内容审核。
3. 评论内容管理。
4. 话题分类维护。
5. AI 回复记录查看。
6. 用户、社区和情绪数据统计。

### 4.4 产品边界

心屿系统不定位为医疗诊断工具，也不替代专业心理咨询服务。

系统更适合定位为：

1. 情绪记录工具。
2. AI 陪伴式反馈工具。
3. 匿名表达社区。
4. 个人情绪趋势观察工具。

因此，后续涉及风险提示、敏感内容识别和严重情绪危机处理时，应采用谨慎设计，避免给出过度专业化判断。

## 5. 技术实现总结

### 5.1 后端技术总结

后端采用 Spring Boot 作为主要开发框架，使用 MyBatis-Plus 操作 MySQL 数据库。

后端核心设计包括：

1. 使用 Controller 接收前端请求。
2. 使用 Service 封装业务逻辑。
3. 使用 Mapper 访问数据库。
4. 使用 Result 统一接口返回结构。
5. 使用 GlobalExceptionHandler 统一异常处理。
6. 使用 BCrypt 对密码进行加密。
7. 使用 JWT 生成和解析登录 token。
8. 使用拦截器或统一校验逻辑保护需要登录的接口。

### 5.2 前端技术总结

前端采用 Vue3 构建页面，使用 axios 进行接口请求并通过 request.js 统一封装，使用 Pinia 管理跨页面状态。

前端核心设计包括：

1. 登录成功后由 user store 统一保存 loginUser 和 token。
2. 请求拦截器自动携带 token，响应拦截器统一处理登录失效。
3. 分层组织：api 层封装接口、stores 层管理状态、composables 层封装通用逻辑，页面只做组装与触发。
4. 使用路由及全局守卫管理页面访问与登录态。
5. 自建设计 token 与基础组件库，统一视觉风格，不依赖第三方 UI 库。
6. 使用全局提示、确认弹窗、加载骨架屏等统一交互反馈。
7. 使用浏览器 DevTools 验证请求、响应和本地存储。

### 5.3 数据库设计总结

数据库设计围绕用户、心情记录、AI 回复、社区内容和统计数据展开。

核心数据关系包括：

```text
user 1 -> N mood_record
user 1 -> N post
user 1 -> N comment
topic 1 -> N post
post 1 -> N comment
post 1 -> N post_like
post 1 -> N favorite
mood_record 1 -> N ai_reply
post 1 -> N ai_reply
```

这种关系设计能够支持个人记录、社区互动和 AI 回复记录的持续扩展。

## 6. 当前系统特点

### 6.1 业务闭环清晰

系统不是孤立实现某一个接口，而是围绕用户登录后的完整行为链路进行设计。

当前基础闭环为：

```text
登录认证
-> 携带 token
-> 新增数据
-> 查询当前用户数据
-> 删除当前用户数据
```

后续扩展闭环为：

```text
记录心情
-> AI 生成回复
-> 用户查看回复
-> 选择是否匿名发布
-> 社区互动
-> 数据统计
```

### 6.2 权限意识明确

系统后端不再依赖前端传递 userId，而是通过 token 解析当前登录用户。

这样可以避免用户通过修改请求参数访问或操作他人数据，提高接口安全性。

### 6.3 返回结构统一

接口统一返回 Result 结构：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

统一返回结构有利于前端统一判断请求结果，也有利于后续异常处理、接口文档维护和前后端协作。

### 6.4 模块扩展空间明确

系统虽然当前处于基础闭环阶段，但已经预留了 AI 回复、匿名社区、后台管理和数据统计等扩展方向。

这使得项目后续可以按阶段持续迭代，而不是推倒重做。

## 11. 风险与应对

### 11.1 AI 回复内容风险

AI 回复可能出现不准确、不合适或过度承诺的问题。

应对方式：

1. 对 AI 回复增加提示词约束。
2. 避免生成医疗诊断类结论。
3. 对高风险情绪内容提供求助提示。
4. 保存 AI 回复记录便于后续追踪。

### 11.2 匿名社区内容风险

匿名表达可能带来违规内容、攻击性内容或垃圾信息。

应对方式：

1. 设置帖子状态。
2. 设置评论状态。
3. 提供管理员审核能力。
4. 增加敏感词检测。
5. 对违规用户进行禁用。

### 11.3 数据安全风险

系统涉及用户情绪内容，属于相对敏感的个人记录。

应对方式：

1. 密码加密存储。
2. 不向前端返回 password。
3. 接口通过 token 获取当前用户。
4. 私密记录默认不公开。
5. 生产环境保护数据库账号和密钥。
