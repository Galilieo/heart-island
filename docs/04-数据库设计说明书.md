# 心屿 AI 情绪记录与匿名互助社区系统

# 04-数据库设计说明书

## 1. 文档目的

本文档用于说明心屿 AI 情绪记录与匿名互助社区系统的数据库设计方案，包括数据建模目标、核心实体、表结构、字段说明、表关系、约束规则、索引设计和后续扩展方向。

数据库设计以产品业务为出发点，同时结合软件工程中的规范化、可维护性、可扩展性和数据安全要求，保证系统能够支撑用户注册登录、情绪记录、AI 回复、匿名社区、评论点赞收藏、后台审核和数据统计等核心功能。

## 2. 设计目标

数据库设计需要满足以下目标：

1. 支撑普通用户注册、登录、个人信息维护和权限识别。
2. 支撑用户长期记录、查询、修改和删除心情记录。
3. 支撑 AI 回复记录的保存、追踪和多业务场景关联。
4. 支撑匿名社区帖子、评论、点赞、收藏和话题分类。
5. 支撑管理员对用户、帖子、评论、话题和 AI 回复记录进行管理。
6. 支撑用户端情绪统计和管理端平台统计。
7. 保证用户隐私数据隔离，避免用户访问他人私有数据。
8. 预留后续 Redis 缓存、ECharts 统计、AI 总结和文件上传等功能扩展空间。

## 3. 数据库命名规范

### 3.1 表命名规范

1. 表名使用小写英文和下划线。
2. 表名尽量使用业务名词，避免过度缩写。
3. 关联表使用业务对象组合命名，例如 post_like。

示例：

```text
user
mood_record
ai_reply
post
comment
post_like
favorite
topic
```

### 3.2 字段命名规范

1. 字段名使用小写英文和下划线。
2. 主键统一使用 id。
3. 外键字段使用 业务名_id，例如 user_id、post_id、topic_id。
4. 时间字段统一使用 create_time、update_time。
5. 状态字段统一使用 status。

### 3.3 通用字段规范

多数业务表建议包含以下通用字段：

| 字段名 | 说明 |
| --- | --- |
| id | 主键 |
| create_time | 创建时间 |
| update_time | 更新时间 |
| status | 状态，适用于需要审核、禁用或逻辑删除的表 |

## 4. 核心实体概览

系统核心实体如下：

| 实体 | 表名 | 说明 |
| --- | --- | --- |
| 用户 | user | 保存普通用户和管理员账号 |

## 5. 实体关系设计

### 5.1 核心关系

系统主要关系如下：

1. 一个用户可以有多条心情记录。
2. 一个用户可以发布多篇帖子。
3. 一个用户可以发表多条评论。
4. 一个用户可以点赞多篇帖子。
5. 一个用户可以收藏多篇帖子。
6. 一个话题下可以包含多篇帖子。
7. 一篇帖子可以有多条评论。
8. 一篇帖子可以被多个用户点赞。
9. 一篇帖子可以被多个用户收藏。
10. 一条心情记录可以关联多条 AI 回复记录。
11. 一篇帖子可以关联多条 AI 回复记录。

### 5.2 关系文本图

```text
user 1 ---- N mood_record
user 1 ---- N post
user 1 ---- N comment
user 1 ---- N post_like
user 1 ---- N favorite

topic 1 ---- N post

post 1 ---- N comment
post 1 ---- N post_like
post 1 ---- N favorite

mood_record 1 ---- N ai_reply
post        1 ---- N ai_reply
```

### 5.3 设计说明

心屿系统的核心不是单表 CRUD，而是围绕用户产生多类内容和互动行为。user 是系统的身份基础，mood_record 是用户私人记录，post 和 comment 是公开社区内容，post_like 和 favorite 是用户互动关系，ai_reply 是系统智能能力的持久化记录。

AI 回复采用 target_type + target_id 的通用关联方式，可以同时关联心情记录、社区帖子或后续聊天消息，避免 AI 回复模块被固定在单一业务表中。

## 6. 用户表 user

### 6.1 表用途

user 表用于保存系统用户信息，包括普通用户和管理员。该表支撑注册、登录、权限识别、用户状态管理和个人资料展示。

### 6.2 字段设计

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 主键，自增 | 用户 id |
| username | varchar(50) | 非空，唯一 | 用户名，用于登录 |
| password | varchar(100) | 非空 | BCrypt 加密后的密码 |
| nickname | varchar(50) | 非空 | 用户昵称 |
| gender | varchar(20) | 可空 | 性别：MALE/FEMALE/OTHER/UNDISCLOSED |
| birthday | date | 可空 | 生日，前端据此推算年龄 |
| bio | varchar(300) | 可空 | 个性签名 |
| city | varchar(50) | 可空 | 所在城市 |
| avatar | varchar(255) | 可空 | 用户头像地址（规划字段） |
| role | varchar(20) | 非空 | 用户角色，USER 或 ADMIN |
| status | tinyint | 非空，默认 1 | 用户状态，1 正常，0 禁用 |
| create_time | datetime | 非空 | 创建时间 |
| update_time | datetime | 可空 | 更新时间 |

### 6.3 字段说明

1. username 用于登录，应保证唯一。
2. password 不保存明文密码，只保存 BCrypt 加密结果。
3. nickname 在社区里展示；其余 gender/birthday/bio/city 均为可选个人资料字段。
4. birthday 存 DATE 类型，年龄由前端动态推算（避免每年更新数据库）。
5. role 用于区分普通用户和管理员。
6. status 用于管理员禁用或启用用户。
7. avatar 用于后续扩展头像上传功能（当前未实装）。

### 6.4 建表 SQL 示例

```sql
CREATE TABLE user (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '用户id',
  username VARCHAR(50) NOT NULL COMMENT '用户名',
  password VARCHAR(100) NOT NULL COMMENT '加密后的密码',
  nickname VARCHAR(50) NOT NULL COMMENT '昵称',
  gender VARCHAR(20) DEFAULT NULL COMMENT '性别：MALE/FEMALE/OTHER/UNDISCLOSED',
  birthday DATE DEFAULT NULL COMMENT '生日（用于推算年龄）',
  bio VARCHAR(300) DEFAULT NULL COMMENT '个性签名',
  city VARCHAR(50) DEFAULT NULL COMMENT '所在城市',
  role VARCHAR(20) NOT NULL DEFAULT 'USER' COMMENT '角色：USER/ADMIN',
  status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1正常，0禁用',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  UNIQUE KEY uk_user_username (username)
) COMMENT='用户表';
```

## 7. 心情记录表 mood_record

### 7.1 表用途

mood_record 表用于保存用户的私人心情记录，是系统用户端的核心业务表。

### 7.2 字段设计

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 主键，自增 | 心情记录 id |
| user_id | bigint | 非空 | 所属用户 id |
| mood_type | varchar(30) | 非空 | 心情类型，如开心、平静、焦虑、难过 |
| content | text | 非空 | 心情内容 |
| ai_reply | text | 可空 | AI 回复内容，便于快速展示 |
| create_time | datetime | 非空 | 创建时间 |
| update_time | datetime | 可空 | 更新时间 |

### 7.3 字段说明

1. user_id 表示记录所属用户。
2. mood_type 用于分类和筛选。
3. content 保存用户输入的心情内容。
4. ai_reply 可以保存最近一次 AI 回复，完整 AI 调用历史保存在 ai_reply 表。
5. 当前用户端版本暂未实现 mood_score 和 is_public 字段，后续做更细粒度情绪评分或公开到社区时再扩展。

### 7.4 建表 SQL 示例

```sql
CREATE TABLE mood_record (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '心情记录id',
  user_id BIGINT NOT NULL COMMENT '用户id',
  mood_type VARCHAR(30) NOT NULL COMMENT '心情类型',
  content TEXT NOT NULL COMMENT '心情内容',
  ai_reply TEXT DEFAULT NULL COMMENT 'AI回复内容',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  KEY idx_mood_user_time (user_id, create_time),
  KEY idx_mood_user_type (user_id, mood_type)
) COMMENT='心情记录表';
```

## 8. AI 回复表 ai_reply

### 8.1 表用途

ai_reply 表用于保存 AI 生成内容和调用记录。该表支撑心情记录回复、帖子回复、情绪总结和后续 AI 调用审查。

### 8.2 字段设计

| 字段名 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 主键，自增 | AI 回复 id |
| user_id | bigint | 非空 | 用户 id |
| target_type | varchar(50) | 非空 | 关联业务类型 |
| target_id | bigint | 非空 | 关联业务 id |
| prompt | text | 可空 | 发送给 AI 的提示词或用户输入 |
| reply_content | text | 可空 | AI 回复内容 |
| model_name | varchar(100) | 可空 | 模型名称 |
| status | tinyint | 非空，默认 1 | 状态，1 成功，0 失败 |
| error_message | text | 可空 | 失败原因 |
| create_time | datetime | 非空 | 创建时间 |

### 8.3 target_type 取值

| 值 | 说明 |
| --- | --- |
| MOOD_RECORD | 心情记录 |
| POST | 社区帖子 |
| CHAT | 聊天消息，预留 |
| SUMMARY | 情绪总结，预留 |

### 8.4 设计说明

AI 回复不直接绑定某一张业务表，而是通过 target_type 和 target_id 建立通用关联。这样同一个 AI 回复模块可以服务不同业务场景，提高扩展性。

### 8.5 建表 SQL 示例

```sql
CREATE TABLE ai_reply (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT 'AI回复id',
  user_id BIGINT NOT NULL COMMENT '用户id',
  target_type VARCHAR(50) NOT NULL COMMENT '关联业务类型',
  target_id BIGINT NOT NULL COMMENT '关联业务id',
  prompt TEXT DEFAULT NULL COMMENT '提示词或用户输入',
  reply_content TEXT DEFAULT NULL COMMENT 'AI回复内容',
  model_name VARCHAR(100) DEFAULT NULL COMMENT '模型名称',
  status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：1成功，0失败',
  error_message TEXT DEFAULT NULL COMMENT '错误信息',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  KEY idx_ai_user_time (user_id, create_time),
  KEY idx_ai_target (target_type, target_id),
  KEY idx_ai_status_time (status, create_time)
) COMMENT='AI回复记录表';
```

### 8.6 当前实现说明

当前代码中，AI 回复主要服务心情记录场景：

1. `target_type` 固定写入 `MOOD_RECORD`。
2. `target_id` 写入对应的 `mood_record.id`。
3. `model_name` 当前保存配置中的 `deepseek-v4-flash`。
4. `prompt` 保存发送给 AI 的系统提示词和用户输入内容。
5. `reply_content` 保存 AI 回复正文；AI 调用失败时保存默认陪伴文案。
6. `status` 为 1 表示调用成功，为 0 表示调用失败。
7. `error_message` 保存 AI 调用失败时的异常信息。

## 14. 约束设计

### 14.1 唯一约束

| 表名 | 字段 | 说明 |
| --- | --- | --- |
| user | username | 用户名唯一 |
| topic | name | 话题名称唯一 |
| post_like | post_id + user_id | 同一用户不能重复点赞同一帖子 |
| favorite | post_id + user_id | 同一用户不能重复收藏同一帖子 |

### 14.2 业务约束

1. 用户密码必须加密存储。
2. 用户只能查询、修改、删除自己的心情记录。
3. 用户只能删除自己的帖子和评论，管理员除外。
4. 帖子和评论需要根据 status 判断是否展示。
5. 禁用用户不能发布帖子、评论、点赞和收藏。
6. AI 回复必须关联具体业务对象。

### 14.3 外键说明

本项目可在业务层维护关联关系，不强制使用数据库外键。这样可以减少开发阶段因删除测试数据导致的外键约束问题，也方便后续做逻辑删除和状态流转。

如果项目进入更严格的数据一致性阶段，可以考虑补充数据库外键约束。

## 15. 索引设计

### 15.1 索引设计原则

1. 高频查询字段需要建立索引。
2. 列表查询常用的 user_id、topic_id、status、create_time 应重点考虑。
3. 联合索引应匹配主要查询条件和排序方式。
4. 唯一业务约束使用唯一索引保证。

## 16. 数据安全设计

### 16.1 密码安全

user.password 字段只保存 BCrypt 加密后的密码，不保存明文密码。

### 16.2 用户隐私

mood_record 默认是用户私人记录，查询时必须根据 token 中的 userId 过滤。前端传入的 userId 不能作为数据权限依据。

### 16.3 敏感数据返回

后端接口返回用户信息时使用 UserVO，不直接返回 user 表完整记录，避免 password 等字段暴露给前端。

### 16.4 社区匿名

post.anonymous 字段用于控制前端是否展示真实用户信息。即使帖子匿名，数据库仍保存真实 user_id，方便用户本人管理内容和管理员进行内容治理。

## 17. 数据统计支持

### 17.1 用户端统计

mood_record 表支持以下统计：

1. 按心情类型统计占比。
2. 统计用户记录总数。
3. 统计最近 7 天或 30 天记录数量。
4. 当前版本已实现最近 7 天心情类型分布和最近一次心情统计。

### 17.2 管理端统计

user、post、comment、ai_reply 表支持以下统计：

1. 用户总数。
2. 今日新增用户数。
3. 帖子总数。
4. 今日新增帖子数。
5. 评论总数。
6. AI 调用次数。
7. 热门话题排行。
8. 高互动帖子排行。

## 19. 数据库设计结论

本系统数据库围绕用户、心情记录、AI 回复、匿名社区和后台管理进行设计。user 表提供身份基础，mood_record 表承载私人情绪记录，ai_reply 表承载智能回复能力，post、comment、post_like、favorite 和 topic 表支撑匿名社区互动。

整体设计既能支持当前最小闭环，也能支撑后续 AI 回复、社区互动、后台审核、数据统计和缓存优化等扩展需求。通过唯一约束、联合索引、状态字段和用户数据隔离规则，数据库结构具备较好的规范性、可维护性和扩展性。
